## 修正後的架構

```
                        Workflow Instance（單一實例貫穿全程）
                        ┌─────────────────────────────────────┐
                        │ businessId = workOrder.id           │
                        │                                     │
                        │ step 1: 派工審核 (甲方)              │
                        │   - 申請人填單                       │
                        │   - 主管審核      ← REJECT 退回填單  │
                        │   - 核准派工                         │
                        │         │                            │
                        │         ▼                            │
                        │ step 2: 施工執行 (乙方) ← ★ 新觀念   │
                        │   ├─ 指派 (PENDING → ASSIGNED)      │
                        │   ├─ 到場打卡 (ASSIGNED → IN_PROGRESS)│
                        │   └─ 完成維修 (IN_PROGRESS → REVIEWING)│
                        │         │                            │
                        │         ▼                            │
                        │ step 3: 完工覆核 (甲方)              │
                        │   ├─ 覆核通過 → step 4               │
                        │   └─ 駁回退回施工 → step 2           │
                        │                                     │
                        │ step 4: 結案 (END)                   │
                        └─────────────────────────────────────┘
```

關鍵改變：**Work Order 不再是獨立的狀態機，而是 Workflow Instance 中一個特殊 step 的內部狀態**。

- Workflow Engine 的 step `施工執行` 是一個 `type=normal` 的 step
- 但這個 step 的 assignee 不是一個人去「簽核」，而是觸發工單操作階段
- 工單內部的細部狀態（PENDING → ASSIGNED → IN_PROGRESS）由 Work Order Service 管理
- 當 `complete()` 被呼叫，才回 call workflow engine 完成這個 step，推進到下一步

## 這樣的好處

| 面向 | 之前（workflow 只包 REVIEWING） | 現在（workflow 包全程） |
|---|---|---|
| **流程可視化** | 只看得到覆核 | 看得到派工 → 施工 → 覆核完整鏈 |
| **駁回流程** | 只能退回 REVIEWING | 可從覆核駁回「施工執行」，要求重工 |
| **SLA 計算** | 只算覆核時間 | 可算每階段的時間（審核幾天、施工幾天） |
| **代理人** | 無關 | 甲方主管可設代理人代審 |
