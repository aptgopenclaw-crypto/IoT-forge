好問題。我徹底研究了 workflow 模組，以下是分析：

---

## Workflow Engine 能做什麼

這是一套**通用簽核引擎**，核心設計是：

```
定義 (workflow_definitions)    實例 (workflow_instances)     歷程 (workflow_step_logs)
┌─────────────────────────┐    ┌────────────────────┐       ┌──────────────────┐
│ code: "work_order"     │    │ businessId: "WO-1" │       │ step: 派工審核   │
│ steps (JSONB):         │    │ currentStepId: ... │       │ action: APPROVE  │
│  派工審核 → 主管覆核     │    │ status: IN_PROGRESS│       │ assignee: userA  │
│  主管覆核 → 財會審核     │    │ contextJson: {...} │  ...  └──────────────────┘
│  財會審核 → END         │    └────────────────────┘
└─────────────────────────┘
```

每個 step 支援：`APPROVE`（通過到下一步）、`REJECT`（退回指定步驟）、`RESUBMIT`（重送）

---

## 工單生命週期 vs Workflow Engine 比對

```
工單狀態機                              Workflow Engine
─────────────────                      ────────────────
PENDING ──指派──→ ASSIGNED               ❌ 沒有「指派」概念
ASSIGNED ─到場打卡─→ IN_PROGRESS          ❌ 沒有「執行中」概念
IN_PROGRESS ─完成維修─→ REVIEWING         ❌ 包含業務邏輯（GPS、費用、故障原因）
REVIEWING ─覆核通過─→ COMPLETED           ✅ APPROVE 完全吻合
REVIEWING ─覆核駁回─→ REJECTED            ✅ REJECT 完全吻合
COMPLETED/REJECTED ─結案─→ CLOSED        ❌ 沒有「結案」概念（但可以搭 Event）
```

## 結論：部分適用，建議混合架構

### 可以交給 Workflow Engine 的部分 ✅

**覆核流程（REVIEWING → COMPLETED / REJECTED）** 完全吻合，可以定義成：

```json
{
  "initial_step": "review",
  "steps": [
    { "id": "review",  "name": "主管覆核", "type": "normal", "next": "end",
      "reject_target": "review", "role_code": "ROLE_WORK_ORDER_REVIEWER" },
    { "id": "end",     "name": "覆核完成", "type": "end" }
  ]
}
```

### 不適合交給 Workflow Engine 的部分 ❌

**操作型步驟（PENDING → ASSIGNED → IN_PROGRESS → REVIEWING）** 含大量領域邏輯：

| 步驟 | 特殊邏輯 |
|---|---|
| 指派 | 需驗證 assignee 是否存在 |
| 到場打卡 | GPS 座標不可為 null |
| 完成維修 | remark 不可空白、cost 不可為負、>10 萬觸發高額審核 |

這些硬用 workflow 去兜反而會把 domain logic 散落到 event listener 裡，更難維護。

---

### 建議架構

```
工單 Service                          Workflow Engine
─────────────────                     ────────────────

create()                    ──→  workflowEngine.start("work_order", workOrderId, ...)
assign()                    ──→  直接更新工單狀態 (PENDING → ASSIGNED)
startWork() (GPS 打卡)      ──→  直接更新 (ASSIGNED → IN_PROGRESS)
complete() (含 cost 驗證)   ──→  直接更新 (IN_PROGRESS → REVIEWING)
                                           ↑
                                           此時自動觸發 workflow 的覆核步驟
                                           (靠 Spring Event 或直接呼叫)

approve() / reject()        ──→  workflowEngine.approve() / reject()
                                  完成時 Spring Event 接回來更新工單狀態

close()                     ──→  工單自己更新狀態 (已無關 workflow)
```

**關鍵設計**：`WorkflowInstanceEntity.businessType = "WORK_ORDER"`，業務 ID 綁工單 ID。覆核完成時透過 `WorkflowStepCompletedEvent` 事件監聽器回寫 `WorkOrder.status = CLOSED`。
