## 後端已有 vs 前端有實作的

| 操作 | 後端 API | 前端畫面 |
|---|---|---|
| 列表查詢 | `GET /auth/work-orders` | ✅ 篩選 + 分頁表格 |
| 檢視明細 | `GET /auth/work-orders/{id}` | ❌ 按鈕存在但**無 handler** |
| 建立工單 | `POST /auth/work-orders` | ❌ **沒有新增按鈕** |
| 指派 | `POST .../{id}/assign` | ❌ 沒有 |
| 到場打卡 | `POST .../{id}/start` | ❌ 沒有 |
| 完成維修 | `POST .../{id}/complete` | ❌ 沒有 |
| 覆核通過 | `POST .../{id}/approve` | ❌ 沒有 |
| 覆核駁回 | `POST .../{id}/reject` | ❌ 沒有 |
| 結案 | `POST .../{id}/close` | ✅ 僅 COMPLETED/REJECTED 可結案 |
| Timeline 歷程 | `GET .../{id}/timeline` | ❌ 沒有 |

## 目前畫面只有兩件事能做

```
工單表格 ── 檢視 (無作用)
         └── 結案 (僅 COMPLETED / REJECTED 狀態)
```

整個工單生命週期的操作（指派→到場→完成→覆核）在前端全部是空白的。

---

## 建議的前端操作流程

以目前工單的生命週期，合理的操作行為是：

```
清單頁                          操作 dialog / drawer
══════════                      ════════════════════

[+ 新增工單]                    → 彈出表單填 deviceId、orderType、description
                                → 建立後 status = PENDING

[指派]  (PENDING 狀態)          → 彈出選擇使用者
                                → 送出後 status = ASSIGNED

[到場打卡] (ASSIGNED 狀態)      → 取得 GPS 座標 (或手動輸入)
                                → 送出後 status = IN_PROGRESS

[完成維修] (IN_PROGRESS 狀態)   → 彈出表單填 remark、faultCause、repairCost
                                → 送出後 status = REVIEWING

[覆核通過/駁回] (REVIEWING 狀態) → 通過 → COMPLETED
                                 → 駁回 → 需填理由 → REJECTED

[結案] (COMPLETED/REJECTED 狀態) → 確認後 → CLOSED

[檢視]                           → Drawer 顯示完整工單資訊 + Timeline 歷程
```

每個操作按鈕依照目前 `row.status` 動態顯示/隱藏，同 DeviceListView 的 pattern。
