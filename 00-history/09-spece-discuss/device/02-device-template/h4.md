# AI Say

## Device Template 需求可行性評估

我完整掃了整個 codebase 後，以下是每個需求的現況與差距分析：

---

### 需求一：DeviceTemplate 作為資料守門員（h1.md）

> 收到 telemetry 時，根據 DeviceTemplate schema 比對格式

| 項目 | 狀態 | 細節 |
|------|:----:|------|
| Entity `DeviceTemplate` | ✅ | 已存在，JSONB `schema`、版本號、tenant 隔離 |
| CRUD API + Frontend | ✅ | `DeviceTemplateController` + DeviceTemplateView.vue（JSON 編輯器） |
| `validateAttributes()` | ⚠️ **只做了 size check**，無實際 schema 欄位驗證 | 需補上依 `fields` 定義逐欄比對的邏輯 |
| Telemetry 接收端 | ❌ **完全沒有** | 無 MQTT endpoint / controller / service 處理 telemetry 上報 |
| Device.java 的 attributes 更新 | ❌ | 有欄位但無地方寫入 |

**可行性：高**。`validateAttributes()` 已有骨架，只需補上：

1. 解析 schema 中的 `fields` 陣列（type / required / min / max）
2. 逐欄位檢查 telemetry payload

---

### 需求二：EventRule 事件規則引擎（h2.md）

> 依據 telemetry 欄位判斷事件是否成立，支援 AND/OR/巢狀

| 項目 | 狀態 | 細節 |
|------|:----:|------|
| `EventRule` Entity | ❌ | 僅在 `h2.md` 有設計文件，未實作 |
| `EventRuleRepository` | ❌ | 不存在 |
| `EventRuleService`（規則引擎） | ❌ | 不存在 |
| `EventRuleController` | ❌ | 不存在 |
| DDL `event_rules` table | ❌ | 不存在 |
| 規則遞迴評估器 | ❌ | `h2.md` 有完整 Java 程式碼，但未寫入專案 |
| Frontend 規則建構器 | ❌ | 不存在 |

**可行性：高**。關鍵邏輯（`evaluateCondition()` 遞迴 AND/OR）已在 `h2.md` 完整設計，直接搬進 `EventRuleService` 即可。需要新增：

- 1 個 Entity + 1 個 Repository + 1 個 Service + 1 個 Controller
- 1 個 Flyway migration（`event_rules` table）
- 1 個前端頁面（規則列表 + JSON 編輯器）

---

### 需求三：EventRule 前端規則建構器（h3.md）

> 視覺化條件構建器，支援 AND/OR 分組、欄位類型感知

| 項目 | 狀態 | 細節 |
|------|:----:|------|
| 規則列表頁 | ❌ | 不存在 |
| 規則編輯頁（條件建構器） | ❌ | 不存在 |
| 即時預覽 | ❌ | 不存在 |
| 規則測試功能 | ❌ | 不存在 |
| 範本載入 | ❌ | 不存在 |

**可行性：中等偏高**，但取決於 UX 深度：

- **簡化版**（P0）：用 JSON 編輯器 + 規則名稱表單，類似 DeviceTemplateView.vue 的模式 → **可行性高**
- **完整版**（P0-P2）：用遞迴元件實作 AND/OR 條件建構器 → **可行但需 2-3 倍工時**
- **第三方套件版**（如 `vue-query-builder`）：可行但需評估 Vue 2/3 相容性

### 整體路線圖建議

```
Phase A: DeviceTemplate 強化（~2 天）
├── 補上 validateAttributes() 的 schema 欄位驗證邏輯
├── 建立 Telemetry ingestion Service（暫存 device.attributes）
└── 新增 V105 migration (device_templates 補 PK id)

Phase B: EventRule 基礎（~3 天）

├── EventRuleController（CRUD）
└── 前端：規則列表頁 + JSON 編輯器

Phase C: 規則引擎整合（~2 天）
├── Telemetry 處理流程插入 event rule 評估
├── 觸發後寫入 DeviceEvent
└── MQTT endpoint 對接

Phase D: 前端強化（可選，~3 天）
├── 視覺化條件建構器（遞迴元件）
├── 規則測試功能
└── 範本載入
```
