## Device Type ↔ Device Template 關聯分析

### 結論：透過共用字串鍵 `device_type` 建立的**邏輯關聯**（非資料庫 FK）

| 面向 | 說明 |
|---|---|
| **關聯方式** | **不是外鍵**，而是透過 `device_type` 字串欄位做邏輯鍵結 |
| **方向** | **Device → DeviceTemplate（多對一）**：多個 Device 可以指向同一個 DeviceTemplate |
| **唯一約束** | `DeviceTemplate` 有 `UNIQUE(tenant_id, device_type)` — 每個租戶下每種設備類型**只能有一個模板** |

---

### 實體對照

**`Device` (`device` 模組)**
```java
// Device.java — 用 device_type 字串來區分設備類型
@Column(name = "device_type", nullable = false, length = 30)
private String deviceType;          // 例如 "STREET_LIGHT"

// 自訂屬性 — 由 DeviceTemplate 的 schema 定義與驗證
@JdbcTypeCode(SqlTypes.JSON)
@Column(name = "attributes", columnDefinition = "jsonb")
private Map<String, Object> attributes;

@Column(name = "format_id")
private Long formatId;              // 預留的關聯欄位（目前未用）
```

**`DeviceTemplate` (`schema` 模組)**
```java
// DeviceTemplate.java — 定義每種 device_type 的 JSON Schema
@Column(name = "device_type", nullable = false, length = 30)
private String deviceType;          // 與 Device.deviceType 對應

// JSON Schema 定義 — 描述 attributes 該有哪些欄位、型別、必填規則
@JdbcTypeCode(SqlTypes.JSON)
@Column(name = "schema", nullable = false, columnDefinition = "jsonb")
private Map<String, Object> schema;
```

### 執行時期如何串連

在 `DeviceService.create()` 建立設備時：

```java
// DeviceService.java
@Transactional
public DeviceResponse create(DeviceRequest request) {
    // ...
    // ★ 關鍵：用 deviceType 字串去查 DeviceTemplate 來驗證 attributes
    deviceTemplateService.validateAttributes(request.getDeviceType(), request.getAttributes());
    // ...
}
```

`DeviceTemplateService` 再根據 `deviceType` 查詢對應的 JSON Schema，用來驗證傳入的 `attributes` 是否合規：

```java
// DeviceTemplateService.java
public Map<String, Object> getSchema(String deviceType) {
    DeviceTemplate template = deviceTemplateRepository.findByDeviceType(deviceType)
        .orElseThrow(() -> new BusinessException(ErrorCode.DEVICE_NOT_FOUND,
                "DeviceType schema not defined: " + deviceType));
    return template.getSchema();
}
```

### API 端點對照

| 模組 | API | 用途 |
|---|---|---|
| `schema` | `GET /v1/auth/device-templates/{deviceType}/schema` | 查某設備類型的 JSON Schema 定義 |
| `schema` | `PUT /v1/auth/device-templates/{deviceType}/schema` | 更新該類型的 Schema |
| `device` | `POST /api/v1/devices` | 建立設備（內部會自動驗證 attributes 是否符合 Schema） |

---

### 一句話總結

> **`DeviceTemplate` 是「設備類型的 Schema 定義」，`Device` 的 `device_type` 指向對應的 `DeviceTemplate`，讓系統能針對不同設備類型動態定義與驗證各自的 JSON 屬性結構。**

這是一種 **Schema-on-demand** 的設計模式：不需要為每種設備類型開新資料表，而是靠 JSONB + JSON Schema 做彈性擴充。目前在程式碼中是**字串鍵結**（沒有 JPA `@ManyToOne` 注解），但 `formatId` 欄位暗示未來可能改為實體 FK 關聯。
