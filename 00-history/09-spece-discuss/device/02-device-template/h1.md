###  實際運作流程

當後端接收到路燈的數據時，會呼叫 `validateAttributes("street_light", telemetryData)`：

1. **取得規則**：系統先去 `device_templates` 表查詢 `deviceType = 'street_light'` 的 Schema。
2. **比對格式**：根據 Schema 中的 `fields` 定義，逐一檢查收到的數據。
3. **決定放行或攔截**：
   * **驗證通過**：數據合法，繼續寫入資料庫或進行後續處理。
   * **驗證失敗**：丟出 `BusinessException(ErrorCode.VALIDATION_ERROR)`，截髒數據。

### 💡 具體情境舉例

假設我們剛剛定義了路燈的 Schema，當設備上報以下數據時：

#### ✅ 情境一：數據合法（順利放行）
```json
{
  "controllerSerial": "C11-202602-016215",
  "brightness": 100,
  "voltage": 233.4,
  "current": 0.3891,
  "power": 87.98,
  "powerFactor": 0.9688
}
```
* **結果**：所有必填欄位都在，且數值都在 `minimum` 和 `maximum` 範圍內（例如亮度 100 在 0-100 之間），**驗證通過**。

#### ❌ 情境二：數據異常（被系統攔截）
```json
{
  "controllerSerial": "C11-202602-016215",
  "brightness": 150, 
  "voltage": "233V", 
  "power": 87.98
}
```
* **結果**：
  1. `brightness` = 150，超過了 Schema 定義的 `maximum: 100` ➡️ **攔截**，回報：`欄位 '亮度(%)' 不可大於 100`。
  2. `voltage` = "233V"，類型是 String 但 Schema 要求 `type: "number"` ➡️ **攔截**，回報：`欄位 '電壓(V)' 應為數字`。
  3. 缺少必填欄位 `current` ➡️ **攔截**，回報：`欄位 '電流(A)' 為必填`。

### ⚠️ 程式碼中的兩個重要「彈性設計」

根據您的 `DeviceTemplateService.java`，有兩個非常貼合 IoT 場景的設計值得注意：

1. **Open Schema（開放式結構）**：
   程式碼邏輯只檢查 Schema 裡有定義的 `key`。如果路燈多傳了一個 Schema 裡沒有的欄位（例如 `"debug_mode": true`），**系統不會報錯**，會直接放行。這能避免設備韌體升級多傳了欄位導致系統崩潰。
2. **向後相容（無 Template 不阻擋）**：
   ```java
   if (schema == null) return; // 無 schema → 自由儲存
   ```
   如果您今天新增了一種新型號的路燈，但**還沒來得及建立 Device Template**，系統收到數據時**不會卡住**，而是直接放行儲存。等您有空時再去補建 Template 即可。
