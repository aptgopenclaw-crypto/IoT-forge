目前程式碼的實際情況是：**沒有強制檢查**。`DeviceService.create()` 雖然有呼叫 `validateAttributes()`，但目前的實作**只檢查 JSONB 大小**，不會驗證 `device_type` 是否已存在於 `DeviceTemplate` 中：

```java
// DeviceTemplateService.java — 目前的 validateAttributes
public void validateAttributes(String deviceType, Map<String, Object> attributes) {
    // ❌ 只檢查 size，沒檢查 deviceType 是否存在
    if (attributes != null && attributes.toString().length() > MAX_JSONB_SIZE) {
        throw new BusinessException(...);
    }
}
```

---

### 這其實是個設計取捨

從架構上來說，**應該要強制** — 原因如下：

1. **資料一致性**：`device_type` 寫錯字（如 `"STREET_LIGHTT"`）不會被發現，之後查 schema 才會炸
2. **前端體驗**：新增設備時 `device_type` 下拉選單應該來自 `DeviceTemplate` 的資料，而非自由輸入
3. **彈性欄位**：沒有 schema 等於不知道 `attributes` 該有哪些欄位，驗證也無從做起

如果決定加上檢查，改動很小 — 在 `DeviceService.create()` 的 `validateAttributes()` 前面補一次 `getSchema()` 即可：

```java
// DeviceService.create() 中加一行：
deviceTemplateService.getSchema(request.getDeviceType());  // ← 找不到就拋錯
deviceTemplateService.validateAttributes(request.getDeviceType(), request.getAttributes());
```

同一個檢查也該補在 `update()` 裡。
