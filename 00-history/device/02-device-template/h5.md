> 有個功能模組設計的討論，device template 和device有關，也和telemetry data是否正確有關，也和事件判斷有關，在功能模組的角度上來看，是否還適合放在device 功能模組，畢竟後來我會有事件功能模組，telemetry模組，通知設定模組等等

# AI Say

好問題。這其實是 bounded context 的邊界劃分問題。讓我分析幾種選項：

---

### 情境分析

DeviceTemplate 被三個模組共用：
```
device ──────────> DeviceTemplate （設備建立時需 schema）
telemetry ───────> DeviceTemplate （驗證上報資料格式）
event-rule ──────> DeviceTemplate （條件欄位來自 schema）
```

問題核心：**三個模組都依賴 device，形成扇入（fan-in）耦合**。

---

### 選項比較

#### 選項 A：留在 device 模組（現狀）

```
device  ─── 對外 export DeviceTemplateService
  ↑
telemetry (import device)
  ↑
event-rule (import device)
```

**優點**：不用搬檔案
**缺點**：
- telemetry 和 event-rule 各自依賴整個 device 模組
- 違反「獨立模組不該相依於實作細節」原則
- `module_exports` 必須把 DeviceTemplateService 列入 public API

#### 選項 B：抽成獨立 schema 模組 ⭐ 推薦

```
schema (device-template)  ── export DeviceTemplateService
  ↑            ↑
device      telemetry    event-rule
（只需 schema 的 schema）
```

**優點**：
- device / telemetry / event-rule 三者**平級**，無誰依賴誰
- DeviceTemplate 的職責純化：只管「某 device type 有哪些合法欄位」
- 未來 MQTT broker 或資料分析模組要讀 schema 也不用繞路

**缺點**：
- 需要搬動 entity / service / repository / controller / migration
- 初期成本較高

#### 選項 C：DeviceTemplate 留在 device，但只暴露介面

```java
// ── device module exports ──
public interface SchemaValidator {
    void validateAttributes(String deviceType, Map<String, Object> attributes);
}

// ── other modules only depend on this interface ──
```

**優點**：Java 層級解耦，不需搬檔案
**缺點**：DB table `device_templates` 還是在 device 的 migration 裡，邏輯上還是同一包

---

### 我的建議

如果確定會長出 **telemetry 模組** 和 **event-rule 模組**，我會選 **選項 B**，原因很具體：

| 面向 | 留在 device | 獨立 schema 模組 |
|------|-----------|----------------|
| module_exports | 需 export 多個 class | 只 export 一個 service |
| module_coupling | telemetry → device (緊) | telemetry → schema (鬆) |
| 擴充性 | device 模組越長越大 | schema 可獨立演進 |
| 未來拆分 microservice | device 變單體瓶頸 | schema 可獨立部署 |

命名可以叫 `device-schema` 或 `iot-schema`，語意上就是「定義 IoT 設備資料格式的規範中心」，比放在 `device` 底下乾淨很多。
