我的建議：

### Telemetry 模組 → 獨立頂層選單

Telemetry（遙測資料）本質上是 IoT 資料匯入與處理的中心，和「設備管理」是不同面向：

```
telemetry    /telemetry
  ├─ 即時資料      (real-time dashboard)
  ├─ 歷史查詢      (time-series query)
  ├─ 資料驗證規則  (validation rules, 參考 schema)
```

### Event Rule 模組 → 獨立頂層選單

事件規則引擎是跨設備的邏輯判斷，也不該被綁在設備管理下：

```
event-rule   /event-rules
  ├─ 事件規則定義  (when device.X > Y → trigger)
  ├─ 觸發記錄      (execution log)
  └─ 通知設定      (notification config)
```

### DeviceTemplate（schema）留在設備管理下

原因：它本質上是**設備類型的資料格式定義**，與設備生命週期綁在一起，放在設備管理下直覺。Telemetry 和 Event Rule 透過 import 方式引用 schema 驗證即可。

```
設備管理      /device
  ├─ 設備列表
  ├─ 設備模板     ← 留在這，因為是設備的屬性定義
  ├─ 工單管理
  ├─ 電力迴路
  └─ 標案契約
```

### 總結

| 模組 | 選單位置 | 原因 |
|---|---|---|
| DeviceTemplate (schema) | 設備管理下 | 設備類型的屬性定義，與設備生命週期綁定 |
| Telemetry | **獨立頂層** `/telemetry` | IoT 資料處理中心，不同 bounded context |
| Event Rule | **獨立頂層** `/event-rules` | 跨設備的事件引擎，獨立演進 |
