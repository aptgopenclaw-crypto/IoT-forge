這個 `AuditAsyncConfig` 類別是一個 **Spring 非同步執行緒池配置類（Async Thread Pool Configuration）**。

它的**核心目的**是：為系統的「審計日誌寫入」任務建立一個**專屬的、參數經過精心調優的背景執行緒池**，以確保非同步寫入日誌時既能高效運作，又不會拖垮或影響系統的其他核心業務。

以下是該類別的詳細功能與企業級設計亮點解析：

### 1. 核心功能
*   **啟用非同步支援**：透過 `@EnableAsync` 註解，開啟 Spring 的非同步方法執行能力。
*   **定義專屬執行緒池**：透過 `@Bean("auditExecutor")` 註冊了一個名為 `auditExecutor` 的 `TaskExecutor`。這正是前面 `AuditAsyncWriter` 中 `@Async("auditExecutor")` 所指定的執行緒池。

### 2. 關鍵參數設定與設計目的
這個類別最精彩的地方在於它對執行緒池參數的設定，展現了對系統穩定性的深思熟慮：

*   **資源隔離與控制**：
    *   `setCorePoolSize(2)`：核心執行緒數為 2。審計寫入通常不是極度高併發的任務，保留 2 個常駐執行緒足以應付日常流量，節省系統資源。
    *   `setMaxPoolSize(8)`：最大執行緒數為 8。當突發流量來臨時，最多擴展到 8 個執行緒同時寫入資料庫，防止資料庫連線池被審計任務瞬間耗盡。
    *   `setQueueCapacity(500)`：任務佇列容量為 500。當 8 個執行緒都在忙碌時，新的寫入任務會被放入佇列排隊。500 的緩衝區足以吸收短暫的流量尖峰。
*   **極致的防丟失機制（背壓/降級保護）**：
    *   *程式碼細節*：`setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy())`
    *   *目的*：這是企業級架構中非常經典的**拒絕策略**。如果佇列滿了（500個），且執行緒也達到上限（8個），系統會拒絕新任務。`CallerRunsPolicy` 的做法是：**「誰呼叫我，誰就親自執行這個任務」**。
    *   *深層含義*：在審計日誌的場景下，這意味著如果背景寫入真的忙不過來，系統會**讓主業務執行緒（例如 Tomcat 的 Request 執行緒）親自去執行寫入日誌的動作**。這雖然會導致該次 API 的響應時間變長（因為主執行緒被阻塞去寫 DB 了），但**保證了審計日誌絕對不會丟失**。這是一種「寧可犧牲一點效能，也要確保合規數據完整」的穩健設計。
*   **便於除錯與監控**：
    *   `setThreadNamePrefix("audit-async-")`：為執行緒命名。當系統發生死結或需要透過 JStack 分析執行緒狀態時，運維人員可以一眼看出哪些執行緒是在處理審計日誌，而不是混雜在預設的 `pool-1-thread-1` 中。

### 總結
如果把 `AuditAsyncWriter` 比作負責搬運貨物的**工人**，那麼 `AuditAsyncConfig` 就是為這些工人搭建的**專屬倉庫與排班系統**。

它不允許工人去搶佔主業務（如訂單處理）的資源（**隔離**），限制了最多只能有 8 個工人同時幹活（**限流**），並且規定如果貨物堆積太多，就連老闆（**主執行緒**）也要親自下場搬貨（**CallerRunsPolicy 防丟失**），從而完美保障了審計系統的穩定性與數據的完整性。
