這個 `BaseLoggerAspect` 類別是一個基於 Spring AOP（面向切面程式設計）實現的 **API 審計日誌記錄器（Audit Logger Aspect）**。

它的**核心目的**是：在不侵入業務程式碼的情況下，自動拦截並記錄系統中關鍵 API 的呼叫行為，以滿足**安全審計、合規追蹤、問題排查**的需求。

以下是該類別的詳細功能與設計亮點解析：

### 1. 核心功能
*   **基於註解的自動攔截**：透過 `@Around("@annotation(auditEvent)")`，它會自動攔截所有被 `@AuditEvent` 註解標記的方法（通常是 Controller 層的 API）。
*   **全方位上下文收集**：在 API 執行前後，收集豐富的環境與使用者資訊，包含：
    *   **使用者與權限**：租戶 ID (`tenantId`)、使用者 ID、帳號、部門 ID、**代操者 ID** (`impersonatedBy`，用於記錄超級管理員模擬操作的場景)。
    *   **網路與環境**：請求 URI、真實客戶端 IP、User-Agent。
    *   **業務數據**：API 的入參（Payload）、事件類型（由 `@AuditEvent` 定義）。
    *   **執行狀態**：API 執行耗時、錯誤碼（區分業務異常與系統異常）。
*   **異步寫入日誌**：最終透過 `AuditAsyncWriter.saveAsync(...)` 將收集到的審計數據**異步**寫入資料庫或日誌系統，避免阻塞主業務線程，保障 API 響應速度。

### 2. 設計目的與安全/穩定性亮點 (從註解與細節看出)
這段程式碼在設計上考慮了許多企業級應用常見的安全與穩定性痛點：

*   **解決 ThreadLocal 上下文丟失問題**：
    *   *程式碼細節*：`// ThreadLocal values must be captured on the main thread`
    *   *目的*：因為日誌是**異步寫入**的，如果等到異步線程再去讀取 `TenantContext` 或 `SecurityContext`（通常基於 ThreadLocal 實現），會讀不到值。因此，程式碼在**主線程**（AOP 攔截時）就提前把租戶、使用者等資訊抓取出來，再傳遞給異步寫入器。
*   **防止 IP 偽造（安全防護）**：
    *   *程式碼細節*：`// 直接使用 TCP 連線來源 IP，不信任可偽造的 X-Forwarded-For header...`
    *   *目的*：很多系統會讀取 `X-Forwarded-For` 來獲取客戶端 IP，但這容易被惡意用戶偽造。此處選擇直接讀取底層 TCP 連線的 `RemoteAddr`，確保審計日誌中的 IP 是絕對真實的，與限流攔截器 (`RateLimitInterceptor`) 保持安全策略一致。
*   **敏感資料脫敏（資料安全）**：
    *   *程式碼細節*：`PayloadSanitizer.sanitize(pjp.getArgs())`
    *   *目的*：API 的入參可能包含密碼、身份證號、Token 等敏感資訊。在寫入審計日誌前，透過 `PayloadSanitizer` 進行脫敏/清洗，防止敏感資料洩露到日誌系統中。
*   **精確的異常分類與追蹤**：
    *   *程式碼細節*：區分 `BusinessException`（提取具體業務錯誤碼）與一般 `Exception`（統一標記為 `99999` 系統異常）。
    *   *目的*：方便後續在審計後台進行數據分析，快速區分是「用戶操作錯誤/業務校驗失敗」還是「系統底層 Bug」。
*   **支援「代客操作」審計**：
    *   *程式碼細節*：`// [Phase B] 代操者 userId...`
    *   *目的*：在 B2B 或 SaaS 系統中，客服或超管可能需要「代操」用戶帳號。這裡記錄了 `impersonatedBy`，確保審計日誌能精確追溯「是誰（代操者）在什麼租戶下，操作了哪個用戶的資料」。

### 總結
這是一個**設計非常嚴謹的企業級審計切面**。它不僅實現了基本的日誌記錄，還周全地考慮了**異步上下文傳遞、IP 防偽造、資料脫敏、代操追蹤**等進階安全與合規需求，主要應用於 IoT 平台或 SaaS 系統中對核心業務操作（如設備控制、權限變更、數據修改等）的監控與追溯。
