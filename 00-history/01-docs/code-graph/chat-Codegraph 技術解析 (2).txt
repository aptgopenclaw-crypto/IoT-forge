### ASSISTANT
你指的應該是 **TypeScript**（簡稱 TS）。

用最簡單的一句話來解釋：**TypeScript 就是「加上『靜態型別檢查』的 JavaScript」。**

它是由微軟開發的開源程式語言，設計目的是為了解決 JavaScript 在大型專案中難以維護、容易出錯的痛點。

以下為你拆解 TypeScript 的核心概念與價值：

---

### 1. 核心差異：動態型別 vs 靜態型別

要理解 TS，必須先知道 JavaScript (JS) 的特性。JS 是「動態型別」，這意味著變數的型別是在**程式執行時**才確定的，這雖然寫起來很自由，但很容易在執行時才爆錯。

**❌ JavaScript 的痛點（執行時才出錯）：**
```javascript
function add(a, b) {
  return a + b;
}

// 寫程式時不會有警告
add(5, 10);      // 結果是 15 (正常)
add("5", 10);    // 結果是 "510" (字串串接，可能不是你想要的！)
add(null, 10);   // 結果是 10 (隱含型別轉換，容易產生難以追蹤的 Bug)
```

**✅ TypeScript 的解法（寫程式時就警告）：**
```typescript
// 明確規定 a 和 b 必須是 number，回傳值也是 number
function add(a: number, b: number): number {
  return a + b;
}

add(5, 10);      // ✅ 正常
add("5", 10);    // ❌ 編輯器立刻畫紅線警告：型別不符！
add(null, 10);   // ❌ 編輯器立刻畫紅線警告：不能傳入 null！
```

---

### 2. TypeScript 的運作原理

瀏覽器或 Node.js **並不認識** TypeScript。TS 的運作流程如下：

1. **開發階段**：你寫 `.ts` 檔案，享受型別檢查和編輯器提示。
2. **編譯階段**：使用 TypeScript 編譯器（`tsc`），將 `.ts` 轉換成純 `.js` 檔案。這個過程會把「型別標記」全部拔掉（稱為型別擦除）。
3. **執行階段**：瀏覽器或 Node.js 執行編譯後產生的 `.js` 檔案。

**結論：TypeScript 最終還是會變成 JavaScript，它只是在開發過程中幫你「把關」的工具。**

---

### 3. 學習 TypeScript 能帶來什麼具體好處？

根據你「功能目標導向」的思維，學習 TS 能達成以下具體成果：

*   **提早抓出 Bug（降低除錯成本）**：把「執行時才會發現的錯誤」提前到「寫程式時」就發現。對於像 `codegraph` 這種需要處理複雜 AST（抽象語法樹）和圖譜資料的專案，型別檢查能避免 80% 的低級錯誤。
*   **極致的編輯器體驗（提升開發效率）**：因為有型別定義，VS Code 等編輯器能提供極其精準的程式碼自動完成（IntelliSense）。你不需要去翻文件看某個物件有哪些屬性，輸入 `.` 就會自動列出來。
*   **重構程式碼不崩潰（降低維護成本）**：在大型專案中，如果你想把一個變數 `userName` 改名為 `userFullName`，TS 會瞬間幫你找出所有用到這個變數的地方，確保沒有遺漏。
*   **自帶文件（降低溝通成本）**：函數的參數和回傳值型別，就是最好的即時文件，團隊協作時不需要靠猜的。

---

### 4. 什麼情況下「不需要」 TypeScript？

雖然 TS 很好，但它也有成本（需要編譯、初期設定較繁瑣、需要學習型別系統）。以下情況用純 JS 即可：
*   非常簡單的腳本（例如幾十行的自動化小工具）。
*   極度追求開發速度、不打算長期維護的一次性原型（Prototype）。

### 1. React + TypeScript (天生一對)
React 的函數式編程思維與 TypeScript 的靜態型別系統結合得非常完美。

*   **現狀**：React 官方文件現在已經全面改用 TypeScript 進行教學。在業界，中大型的 React 專案幾乎 100% 使用 TypeScript。
*   **寫法**：React 的 JSX 語法配合 TS 時，檔案副檔名會從 `.jsx` 變成 **`.tsx`**。
*   **優勢**：
    *   **Props 型別檢查**：可以明確定義元件接收的參數（Props）型別，傳錯資料編輯器會立刻警告。

```tsx
// 明確定義 Props 的型別

  label: string;
  onClick: () => void;
  isDisabled?: boolean; // ? 代表可選
}

// 編輯器會嚴格檢查傳入的 props 是否符合 ButtonProps
const Button: React.FC<ButtonProps> = ({ label, onClick, isDisabled }) => {
  return (
    <button onClick={onClick} disabled={isDisabled}>
      {label}
    </button>
  );
};
```

### 2. Vue + TypeScript (Vue 3 完美融合)
如果你聽過「Vue 對 TypeScript 支援不好」，那是 **Vue 2 時代**的舊觀念了。

*   **現狀**：**Vue 3 的底層源碼就是用 TypeScript 重寫的**。因此，Vue 3 對 TS 提供了原生的、極佳的支援。
*   **寫法**：在 `.vue` 單檔案元件中，只需在 `<script>` 標籤加上 `lang="ts"` 即可（特別是搭配 `<script setup>` 語法糖）。
*   **優勢**：
    *   **Composition API**：Vue 3 的組合式 API 讓型別推斷變得非常直觀，不再像 Vue 2 的 Options API 那樣需要寫一堆複雜的型別裝飾器。
    *   **極致的編輯器體驗**：搭配 Volar 插件，Vue 3 + TS 的模板（Template）內也能享受精準的型別提示。

**範例 (Vue 3 + TS)：**
```vue
<script setup lang="ts">
import { ref } from 'vue'

// 明確定義 count 是 number 型別
const count = ref<number>(0)

// 定義 Props 型別 (Vue 3.3+ 語法)
defineProps<{
  title: string
  count?: number
}>()
</script>

<template>
  <!-- 編輯器會知道 title 是字串，count 是數字 -->
  <h1>{{ title }}</h1>
  <button @click="count++">Count is: {{ count }}</button>
</template>
```

### 2. 檢查專案根目錄的設定檔（最關鍵）

### 3. 檢查 `package.json` 的依賴套件
打開專案根目錄的 `package.json` 檔案，在 `dependencies` 或 `devDependencies` 區塊中搜尋以下關鍵字：

*   **✅ 有引入 TS**：你會看到以下套件：
    *   `"typescript"`：TS 核心編譯器。
    *   `"vue-tsc"`：Vue 專屬的 TS 型別檢查工具（用於檢查 `.vue` 檔案中的 template 型別）。
    *   `"@types/node"` 或 `"@types/..."`：Node.js 或其他套件的 TS 型別定義。

### 4. 實戰測試：執行型別檢查指令（最嚴謹）
如果前面的檔案都有，你可以直接在終端機（Terminal）執行型別檢查指令，看看系統是否真的能正確解析 TS：

在專案根目錄下執行：
```bash
npx vue-tsc --noEmit
```
*(註：`--noEmit` 代表只檢查型別，不輸出編譯後的 JS 檔案)*

*   **✅ 有引入且設定正確**：終端機會顯示檢查進度，如果程式碼沒寫錯，最後會安靜地結束（沒有錯誤訊息）。
*   **❌ 沒有引入或設定錯誤**：終端機會報錯，提示找不到 `vue-tsc` 指令，或是 `tsconfig.json` 設定有誤。
*(如果是用 Vite 建立的標準 TS 專案，也可以直接執行 `npm run type-check` 或 `npm run build` 來觸發檢查)*

Tree-sitter 是什麼

### ASSISTANT
用最白話的一句話來解釋：**Tree-sitter 是一個能把「人類寫的程式碼」快速轉換成「電腦能理解的樹狀結構（AST）」的超強工具。**

它最初是由 GitHub 開發，現在已經成為現代程式碼編輯器、AI 輔助工具、語法分析工具的**底層核心基礎建設**。

以下為你拆解 Tree-sitter 的核心概念與強大之處：

---

### 1. 它到底在做什麼？（從原始碼到 AST）

當我們寫程式時，對人類來說是「文字」，但電腦需要理解它的「結構」。Tree-sitter 的工作就是把文字變成 **AST（抽象語法樹）**。

**範例：一段簡單的 JavaScript**
```javascript
function add(a, b) {
  return a + b;
}
```

**Tree-sitter 解析後會變成這樣的樹狀結構（S-expression 格式）：**
```text
(function_declaration
  name: (identifier)
  parameters: (formal_parameters
    (identifier)
    (identifier))
  body: (statement_block
    (return_statement
      (binary_expression
        left: (identifier)
        right: (identifier)))))
```
*看到沒？它把 `function` 識別為函數宣告，把 `a, b` 識別為參數，把 `a + b` 識別為二元運算式。這就是程式碼的「骨架」。*

---

### 2. 為什麼 Tree-sitter 這麼受歡迎？（三大核心優勢）

在 Tree-sitter 出現之前，要解析程式碼通常用傳統的編譯器（如 Babel, ESLint 底層），但它們有幾個痛點。Tree-sitter 完美解決了這些問題：

#### 🚀 優勢一：極致的速度（增量解析 Incremental Parsing）
*   **傳統工具**：你改了 1 行程式碼，它要把整個幾萬行的檔案重新解析一次，非常慢。
*   **Tree-sitter**：它會記住之前的解析結果。當你修改 1 行時，它**只重新解析那 1 行及其受影響的父節點**。這讓它能在你打字的同時（毫秒級）即時更新語法樹。

#### 🛡️ 優勢二：超強容錯性（Error Tolerance）
*   **傳統工具**：如果你的程式碼有語法錯誤（例如少了一個括號 `}`），傳統解析器會直接崩潰或拒絕工作，導致語法高亮失效。
*   **Tree-sitter**：即使你的程式碼寫到一半、充滿語法錯誤，它依然會盡力解析，並把錯誤的部分標記為 `ERROR` 節點，**整棵樹依然保持完整**。這對於編輯器在開發者打字時提供即時回饋至關重要。

#### 🌍 優勢三：多語言與跨平台（WASM 支援）
*   它使用一套統一的語法來定義各種語言的規則（Grammar）。目前社群已經實作了 100+ 種語言的解析器。
*   它的底層是 C 語言，但可以輕鬆編譯成 **WebAssembly (WASM)**。這意味著你可以直接在瀏覽器或 Node.js 環境中執行它，不需要依賴系統底層。

---

### 3. 實際應用場景（你每天都在用，只是不知道）

Tree-sitter 已經無所不在，以下工具底層都重度依賴它：

1.  **程式碼編輯器語法高亮**：
    *   **GitHub**：你在 GitHub 上看程式碼的漂亮語法高亮，就是 Tree-sitter 做的。
    *   **Neovim / Helix**：現代終端編輯器的語法高亮與程式碼折疊核心。
    *   **VS Code**：雖然 VS Code 傳統上用 TextMate，但現在也全面轉向支援 Tree-sitter（特別是透過 extensions）。
2.  **AI 輔助編程工具**：
    *   就像你之前看的 **`codegraph`**，AI 需要理解你的程式碼結構（誰呼叫了誰、有哪些類別），Tree-sitter 是提取這些結構最快、最準的工具。
    *   Cursor、GitHub Copilot 底層也有使用類似技術來理解上下文。
3.  **現代 Linter / Formatter**：

---

### 1. 為什麼「Tree-sitter + AI」比傳統模板更強？

假設你的 IoT 模板專案中，有一個 `main.ts` 需要初始化各種感測器模組：

**傳統模板引擎 (Handlebars) 的做法：**
```javascript
// 模板寫起來很痛苦，且容易破壞縮排
{{#each sensors}}
import { init{{this.name}} } from './modules/{{this.name}}';
{{/each}}

async function main() {
  {{#each sensors}}
  await init{{this.name}}();
  {{/each}}
}
```
*痛點：只能做死板的文字替換。如果 `main.ts` 裡面有複雜的條件判斷、錯誤處理，模板會變得極其難以維護。*

**Tree-sitter + AI 的做法：**
AI 不需要看死板的模板。AI 透過 Tree-sitter 讀取 `main.ts` 的 AST，**理解**這是一段「函數宣告」，裡面有「呼叫表達式」。AI 可以直接決定：「我要在 `main` 函數的 `body` 節點中，插入一個新的 `await initDHT11()` 節點」，然後自動生成完美縮排的程式碼。

---

### 2. 實作架構：如何打造你的「IoT AI 架構師」

要實現這個想法，你可以設計一個 AI Agent 工作流，分為四個階段：

#### 階段一：模板標記與解析 (Template Parsing)
1. 準備一個標準的 IoT 模板專案。
2. 使用 Tree-sitter 將模板專案中的關鍵檔案（如 `main.ts`, `config.json`, `router.ts`）解析為 AST。
3. **提取結構 (Schema Extraction)**：讓 AI 或腳本分析 AST，提取出「可組合的插槽」。例如：「這個專案允許在 `modules` 陣列中插入新的感測器物件」。

#### 階段二：需求理解與決策 (AI Reasoning)
1. 你輸入自然語言需求：「我要開一個『智慧溫室』專案，需要 DHT11 溫濕度感測器、土壤濕度感測器，並使用 MQTT 協議上傳到 AWS IoT Core。」
2. AI Agent 接收需求，並結合階段一提取的「模板結構資訊（AST 摘要）」。
3. AI 進行推理：「我需要引入 `dht11` 和 `soil_moisture` 模組，修改 `config.json` 的 `sensors` 陣列，並在 `main.ts` 中初始化它們。」

#### 階段三：程式碼生成與修改 (Code Generation / Modification)
這裡有兩條技術路線可選：

*   **路線 A：AI 直接生成 (Generative)**
    AI 根據模板的 AST 結構作為 Few-shot 範例，直接生成新的 `main.ts` 或模組檔案。
*   **路線 B：結構化替換 (Codemod / AST Editing)**
    AI 輸出具體的「修改指令」（例如：在 `main` 函數的第 3 行插入 `initDHT11()`），然後由底層工具執行修改。

#### 階段四：語法驗證 (Validation)
生成程式碼後，**再次使用 Tree-sitter** 解析新生成的檔案。如果 Tree-sitter 報錯（出現 `ERROR` 節點），代表 AI 生成的程式碼有語法錯誤，Agent 會自動捕獲錯誤並要求 AI 重新生成，直到 AST 完全合法。

---

### 3. 關鍵技術工具推薦 (不要只用純 Tree-sitter)

雖然 Tree-sitter 是核心，但直接用它的底層 API 去「修改」程式碼並重新輸出（Unparse）會非常痛苦（因為要處理空白、縮排、註解等瑣碎細節）。建議使用以下封裝好的工具：

#### 🌟 核心推薦：`ast-grep` (sg)
這是目前基於 Tree-sitter 最強大的**結構化搜尋與替換**工具。
*   **用法**：你可以寫類似程式碼片段的規則（YAML 格式），它會精準匹配 AST 並進行替換。
*   **結合 AI**：你可以讓 AI 生成 `ast-grep` 的 YAML 規則，然後執行 `sg scan --rewrite` 來批量修改模板專案。這比讓 AI 直接改文字安全 100 倍。

#### 🌟 輔助推薦：`GritQL` 或 `jscodeshift`
*   如果你主要寫 JS/TS，`jscodeshift` (Meta 開發) 是老牌但強大的 AST 轉換工具。
*   `GritQL` 是新一代的程式碼搜尋與修改語言，對 AI 非常友善。

#### 🌟 AI Agent 框架：`LangChain` 或 `MCP (Model Context Protocol)`
*   你可以把 Tree-sitter 的查詢能力封裝成 **MCP Tools**。
*   讓 AI Agent 透過 MCP 呼叫 `search_ast`、`insert_code_at_node` 等工具，像操作資料庫一樣操作你的程式碼。

---

### 4. 實戰情境模擬：加入一個新模組

假設你的 IoT 框架是這樣設計的，現在你要用 AI 自動加入 `MQTT` 模組：

1.  **AI 讀取 AST**：AI 透過 Tree-sitter 讀取 `src/index.ts`，找到 `App` 類別的 `constructor` 節點。
2.  **AI 決策**：AI 決定需要在 `constructor` 內部，實例化 `MqttClient` 並注入到 `this.dependencies`。
3.  **AI 生成修改指令 (ast-grep 規則)**：
    ```yaml
    # AI 自動生成的 ast-grep 替換規則
    id: add-mqtt-init
    language: typescript
    rule:
      pattern: constructor() { $$$BODY }
    fix: |
      constructor() {
        this.mqtt = new MqttClient(config.mqtt);
        $$$BODY
      }
    ```
4.  **執行與驗證**：腳本執行替換，然後 Tree-sitter 再次解析 `src/index.ts`，確認沒有 `ERROR` 節點，專案生成完成！

### 💡 給你的落地建議

這個想法完全可行，但為了避免一開始陷入底層 AST 操作的泥沼，建議採取 **「漸進式實作」**：

**Phase 1：AST 作為 AI 的「高級 RAG (檢索增強生成)」**
*   不要讓 AI 去「修改」程式碼。
*   讓 Tree-sitter 把模板專案的「目錄結構、類別定義、函數簽名」提取成結構化的 JSON。
*   把這個 JSON 餵給 AI，讓 AI **直接生成完整的新檔案**（而不是修改舊檔案）。這能避開 AST 編輯的痛點，同時享受語意理解的紅利。

**Phase 2：引入 `ast-grep` 進行微調**
*   當 AI 生成好主要檔案後，對於一些「樣板程式碼（Boilerplate）」（例如：在 `package.json` 加入依賴、在 `index.ts` 加入 import 語句），讓 AI 生成 `ast-grep` 規則來自動注入。

**Phase 3：打造專屬的 MCP Server**
*   將你的 IoT 模組庫、Tree-sitter 解析能力封裝成 MCP Server。
*   未來你只需要在 Cursor 或 Claude Code 中說：「幫我用 IoT 模板建一個智慧農業專案，加入 NPK 感測器」，AI 就能自動呼叫你的工具完成一切。

## 1. SQLite — 零配置的嵌入式資料庫

### 它與 MySQL / PostgreSQL 的差別

| | MySQL / PostgreSQL | SQLite |
|---|---|---|
| **架構** | 需要安裝伺服器，背景持續運行 | **不需要伺服器**，就是一個 `.db` 檔案 |
| **安裝** | 要裝資料庫軟體、開 port、管帳號 | 零安裝，Node.js 內建支援 |
| **適合場景** | 網站後端、多使用者併發 | 嵌入式應用、桌面工具、離線工具 |
| **資料量** | 可處理 TB 級 | 單檔可到數百 GB（對專案程式碼綽綽有餘） |

### 白話比喻

> MySQL 就像一間**銀行**，你要開戶、排隊、透過櫃檯存取。
> SQLite 就像一個**保險箱**，放在你家裡，打開就用，不需要網路。

### 在 codegraph 中的角色

codegraph 的所有資料都存在專案根目錄的 `.codegraph/codegraph.db` 這**一個檔案**裡。不需要啟動資料庫伺服器，不需要網路連線，`git clone` 下來就能用。

## 2. FTS5 — SQLite 內建的全文搜尋引擎

### 為什麼需要全文搜尋？

假設你有一個程式碼庫，存了幾千個函數名稱和程式碼片段。如果你想搜尋 `calculateTax`：

**❌ 傳統 SQL LIKE 查詢（很慢）：**
```sql
-- 逐行掃描，資料量大時會非常慢
SELECT * FROM functions WHERE name LIKE '%calculateTax%';
```
這就像在圖書館裡**一本一本書翻開來看**，找有沒有你要的字。

**✅ FTS5 全文搜尋（極快）：**
```sql
-- 利用反向索引，毫秒級回傳結果
SELECT * FROM functions_fts WHERE functions_fts MATCH 'calculateTax';
```
這就像圖書館有**索引目錄卡**，直接查目錄就知道在哪本書的第幾頁。

### FTS5 的核心原理：反向索引（Inverted Index）

FTS5 會在資料寫入時，自動建立一個「字詞 → 出現在哪些文件」的對照表：

```
"calculate" → [檔案A第3行, 檔案C第12行, 檔案F第45行]
"tax"       → [檔案A第3行, 檔案B第8行]
"sensor"    → [檔案D第1行, 檔案E第22行]
```

所以當你搜尋 `"calculate tax"` 時，它不需要掃描所有資料，而是直接查這個對照表，找出同時包含兩個詞的交集，**毫秒級回傳**。

### 在 codegraph 中的應用

當你在 AI 代理中輸入：「找出所有跟 `溫度計算` 相關的函數」，底層就是透過 FTS5 去搜尋函數名稱、註解、程式碼片段，瞬間找出所有匹配的節點。

---

## 3. 知識圖譜（Knowledge Graph）— 把程式碼變成社交網路

### 什麼是知識圖譜？

知識圖譜就是用 **「節點（Node）」和「邊（Edge）」** 來描述事物之間的關係。

### 在程式碼世界裡的樣子

把程式碼想像成一個社交網路：

```
節點 (Nodes)                    邊 (Edges / 關係)
─────────────────               ─────────────────
📦 Module: auth                 ➡️ imports → 📦 Module: database
📄 File: sensor.ts              ➡️ exports → 📦 Module: utils
⚙️ Function: calculateTax()     ➡️ calls → ⚙️ Function: getRate()
📦 Class: TemperatureSensor     ➡️ extends → 📦 Class: BaseSensor
```

### 在 SQLite 中怎麼存圖譜？

關聯式資料庫不是圖資料庫，但可以用 **「節點表 + 邊表」** 來模擬：

```sql
-- 節點表：存所有的函數、類別、檔案
CREATE TABLE nodes (
  id         INTEGER PRIMARY KEY,
  name       TEXT,           -- 例如 "calculateTax"
  kind       TEXT,           -- 例如 "function", "class", "file"
  file_path  TEXT,           -- 例如 "src/billing/tax.ts"
  line_start INTEGER,
  line_end   INTEGER,
  code_snippet TEXT          -- 程式碼片段（供 FTS5 搜尋）
);

-- 邊表：存關係
CREATE TABLE edges (
  id         INTEGER PRIMARY KEY,
  source_id  INTEGER REFERENCES nodes(id),  -- 誰
  target_id  INTEGER REFERENCES nodes(id),  -- 對誰
  kind       TEXT   -- 關係類型："calls", "imports", "extends"
);
```

### 這樣做的好處：可以回答複雜問題

```sql
-- 問題：calculateTax() 呼叫了哪些函數？
SELECT n.name, n.file_path
FROM nodes n
JOIN edges e ON e.target_id = n.id
JOIN nodes src ON src.id = e.source_id
WHERE src.name = 'calculateTax' AND e.kind = 'calls';

-- 問題：如果修改了 getRate()，哪些函數會受影響？（影響分析）
-- 反向查詢所有 "calls" 指向 getRate 的節點，再遞迴往上找
```

### 場景：建立「模組知識庫」

```
模組知識庫 (modules.db)
│
├── nodes 表：記錄你所有的 IoT 模組
│   ├── "DHT11 溫濕度模組" (kind: sensor)
│   ├── "MQTT 通訊模組" (kind: communication)
│   ├── "PID 控制器模組" (kind: controller)
│   └── "AWS IoT Core 適配器" (kind: adapter)
│
├── edges 表：記錄模組之間的相容性與依賴
│   ├── DHT11 → requires → GPIO 模組
│   ├── MQTT → requires → WiFi 模組
│   └── PID 控制器 → compatible → DHT11
│
└── FTS5 索引：支援模糊搜尋
    └── 搜尋 "溫度" → 找到 DHT11, 熱電偶模組, PID 控制器...
```

**AI Agent 的工作流：**
1. 你說：「我要一個溫室監控專案，需要溫度和土壤濕度」
2. AI 透過 FTS5 搜尋模組庫 → 找到 DHT11 和 SoilMoisture 模組

5. AI 生成完整的 `package.json`、`main.ts`、`config.json`

> **這就是用 SQLite + FTS5 把「你的經驗」變成「AI 可以查詢的知識庫」。**

### 3. 實作上的 3 個關鍵挑戰與解法（避坑指南）

這個想法很棒，但在實作時會遇到以下挑戰，提前知道可以少走彎路：

#### 挑戰一：Token 爆炸（Context Window 限制）
*   **問題**：你的 DB 裡可能有幾十萬行程式碼，AI 的上下文視窗（Context Window）放不下，全塞進去會變得很笨且昂貴。
*   **解法：兩階段檢索 (Two-stage Retrieval)**。
    *   第一階段：只搜尋「函數簽名、註解、摘要」，讓 AI 找出「相關的模組」。
    *   第二階段：確定要修改哪些檔案後，**只把那几个相關檔案的具體實作**餵給 AI 進行生成。

#### 挑戰二：關鍵字搜尋的語意落差
*   **問題**：FTS5 是關鍵字搜尋。如果你搜「溫度」，可能找不到函數名叫 `readDHT11()` 的程式碼。
*   **解法：引入向量搜尋 (Vector Search)**。
    *   在 SQLite 中加入向量擴展（如 `sqlite-vec`），或者使用 Pinecone/Qdrant。
    *   把程式碼轉成 Embedding（向量）。這樣 AI 就能進行「語意搜尋」，搜「溫度」就能精準找到 `readDHT11()`。

#### 挑戰三：程式碼變更的同步
*   **問題**：你手動改了程式碼，但 SQLite DB 沒更新，AI 就會參考到舊的程式碼（幻覺來源）。
*   **解法：自動化同步機制**。
    *   像 `codegraph` 一樣，寫一個 File Watcher 監聽檔案變更，即時更新 DB。
    *   或者在 Git Commit 時，觸發 CI/CD 腳本重新掃描並更新 DB。

### 1. 解決「邊界」問題：提取 API 表面 (API Surface)

AI 不需要知道 `auth` 模組裡面幾千行程式碼是怎麼寫密碼加密的，它只需要知道 **「我該怎麼呼叫它」**。這在 TypeScript 中就是 `.d.ts` 宣告檔的概念。

**Tree-sitter 的任務：**
精準抓取模組的 `export` 語句、`interface`、`type` 和函數簽名（參數與回傳值）。

**存入 SQLite 的設計：**
建立一個 `module_exports` 表，專門存儲邊界資訊。

```sql
CREATE TABLE module_exports (
  id INTEGER PRIMARY KEY,
  module_name TEXT,       -- 例如 'auth'
  export_type TEXT,       -- 'function', 'class', 'interface', 'type', 'const'
  export_name TEXT,       -- 例如 'login', 'UserPayload'
  signature TEXT,         -- 完整的型別簽名，例如 'login(username: string, password: string): Promise<Token>'
  jsdoc TEXT              -- 該介面的 JSDoc 註解
);
```

**AI Agent 的視角：**
當 AI 需要寫登入功能時，它查詢 `SELECT signature, jsdoc FROM module_exports WHERE module_name = 'auth'`。
**結果**：AI 瞬間知道了必須 import `login`，且必須傳入兩個字串，回傳一個 Promise。它**完全不需要猜測參數格式**。

---

### 2. 解決「依賴與耦合」問題：提取依賴圖譜 (Dependency Graph)

模組的邊界不僅是「它提供什麼」，還包含「它需要什麼」。如果 `rbac` 模組底層依賴了 `auth` 模組來獲取當前用戶，AI 在組合時就必須知道這個隱性邊界。

**Tree-sitter 的任務：**
掃描每個模組內所有檔案的 `import` 語句，解析出它依賴了哪些內部模組。

**存入 SQLite 的設計：**
建立一個 `module_dependencies` 邊表（這就是知識圖譜的核心）。

```sql
CREATE TABLE module_dependencies (
  source_module TEXT,  -- 例如 'rbac'
  target_module TEXT,  -- 例如 'auth'
  import_path TEXT     -- 例如 '../auth/userService'
);
```

**AI Agent 的視角：**
當 AI 準備生成 `rbac` 相關程式碼時，它查詢 `SELECT target_module FROM module_dependencies WHERE source_module = 'rbac'`。
**結果**：AI 發現 `rbac` 依賴 `auth`。於是 AI 在生成程式碼時，會**自動且正確地**在檔案開頭加上 `import { getCurrentUser } from '../auth'`，而不是自己瞎編一個 `getUser()`。

---

### 3. 解決「內容」問題：提取邏輯特徵與摘要 (Semantic Context)

AI 會猜測內容，通常是因為它不知道這個模組的「業務邏輯特徵」或「隱藏的坑（Gotchas）」。我們不需要把幾千行實作餵給 AI，而是餵給它「高階摘要」。

**實作方式（Tree-sitter + LLM 輔助）：**
1. 用 Tree-sitter 找出模組的「核心入口函數」或「主要 Class」。
2. 將這些核心程式碼片段（或整個模組的結構）丟給 LLM，讓 LLM 生成一段 **100 字以內的自然語言摘要**，以及 **3 條使用注意事項**。
3. 將這些 AI 生成的摘要存入 SQLite。

**存入 SQLite 的設計：**
```sql
CREATE TABLE module_manifest (
  module_name TEXT PRIMARY KEY,
  description TEXT,      -- 自然語言摘要：「負責處理 JWT 簽發與驗證，並整合 Redis 做黑名單管理」
  gotchas TEXT,          -- 注意事項：「1. 必須先呼叫 initAuth() 初始化 Redis。2. Token 過期時間由 config 模組控制，不可硬編碼。」
  tech_stack TEXT        -- 技術特徵：「使用 jsonwebtoken 套件，非同步操作」
);
```

**AI Agent 的視角：**
AI 在寫程式前，先讀取 `module_manifest`。
**結果**：AI 看到了 `gotchas`，它就會在生成程式碼時，**主動加上** `await initAuth()` 的呼叫，並且不會去硬編碼 Token 時間。這就消除了 AI 「邊寫邊猜」業務邏輯的問題。

### 4. 終極殺手鐧：提供「黃金範例」 (Golden Examples)

AI 最強大的學習方式是 Few-shot prompting（少樣本提示）。與其讓 AI 看冰冷的介面定義，不如直接給它看「你過去寫過最標準的範例」。

**Tree-sitter 的任務：**
在每個模組中，標記一個「標準使用範例」的檔案或函數（例如 `auth/example.usage.ts`，或者在 JSDoc 中標記 `@example`）。Tree-sitter 將這些範例程式碼完整提取出來。

**存入 SQLite 的設計：**
```sql
CREATE TABLE module_examples (
  module_name TEXT,
  example_code TEXT  -- 完整的、可運行的範例程式碼
);
```

**AI Agent 的視角：**
當 AI 要組合 `auth` 和 `rbac` 時，它查詢這兩個模組的 `example_code`。
**結果**：AI 直接參考你過去寫過的、經過驗證的範例，照著它的結構和命名風格來寫新程式碼。這能確保新程式碼 100% 符合你的團隊規範。

### ❌ 為什麼不建議「auth 一筆資料，底下包 JSON」？

如果你把 `auth` 的所有 API 塞進一個 JSON 欄位：
```json
{
  "module": "auth",
  "apis": [
    { "name": "login", "desc": "...", "example": "..." },
    { "name": "refreshToken", "desc": "...", "example": "..." }
  ]
}
```
**會遇到三個致命痛點：**
1. **FTS5 全文檢索失效**：SQLite 的 FTS5 擅長搜尋「純文字欄位」。如果 API 描述混在一個大 JSON 字串裡，搜尋的粒度會變得很粗，很難精準定位到「只搜尋 refreshToken 的描述」。
2. **Token 爆炸**：當 AI 只需要 `login` 的資訊時，你卻把整個 `auth` 模組（包含 20 個 API 的範例）都塞進 Prompt 裡，瞬間浪費大量 Token，且 AI 容易失焦。
3. **更新困難**：如果你只修改了 `refreshToken` 的範例，你需要把整個大 JSON 讀出來、修改、再寫回資料庫，效能較差。

---

### ✅ 正確做法：每個 API 一筆資料（關聯式設計）

在關聯式資料庫（SQLite）中，我們應該把「模組」和「API」拆成兩張表，透過 `module_name` 關聯。這不僅符合資料庫正規化，更是 **AI 精準檢索的最佳實踐**。

#### 1. 模組表 (Module Manifest) - 存「宏觀」資訊
這張表只存 `auth` 這**一整包**的高階資訊。一個模組只有一筆資料。

```sql
CREATE TABLE modules (
  module_name TEXT PRIMARY KEY,  -- 'auth'
  description TEXT,              -- 宏觀描述：「處理 JWT 簽發、驗證與黑名單管理」
  dependencies TEXT,             -- 依賴：'["config", "database"]'
  gotchas TEXT                   -- 全局注意事項：「啟動前必須先呼叫 initAuth()」
);
```

#### 2. API 表 (API Surface) - 存「微觀」資訊
這張表存**每一個 API**。`auth` 模組有幾個 API，這裡就有幾筆資料。

```sql
CREATE TABLE module_apis (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  module_name TEXT,              -- 'auth' (用來關聯 modules 表)
  api_name TEXT,                 -- 'login', 'refreshToken', 'logout'
  api_type TEXT,                 -- 'function', 'class', 'controller'
  signature TEXT,                -- 完整簽名：'login(username: string, password: string): Promise<Token>'
  description TEXT,              -- 這個 API 的詳細描述
  example_code TEXT,             -- 這個 API 的標準使用範例
  
  -- 建立 FTS5 全文檢索索引 (針對 api_name, signature, description)
  UNIQUE(module_name, api_name)
);

-- 建立 FTS5 虛擬表，讓 AI 可以秒搜 API
CREATE VIRTUAL TABLE module_apis_fts USING fts5(
  module_name, 
  api_name, 
  signature, 
  description, 
  content='module_apis', 
  content_rowid='id'
);
```

### 1. 深化你的方案：建立「通用核心概念 (Universal Core Concepts)」

你提到要對不同語言的 AST 做抽象化對應。具體來說，我們需要在 Tree-sitter 和 SQLite 之間，插入一個 **「適配器層 (Adapter Layer)」**，將百家爭鳴的 AST 節點，映射到一套 **「通用語言」**。

**實作範例：**

| 通用概念 (Universal Kind) | TypeScript AST 節點 | Python AST 節點 | C/C++ AST 節點 |
| :--- | :--- | :--- | :--- |
| **`Function`** | `function_declaration`, `arrow_function` | `function_definition` | `function_definition` (包含 `function_declarator`) |
| **`Class`** | `class_declaration` | `class_definition` | `class_specifier` |
| **`Interface/Type`** | `interface_declaration`, `type_alias_declaration` | *(無直接對應，通常用 dataclass 或 TypedDict)* | `struct_specifier`, `union_specifier` |
| **`Import/Include`** | `import_statement`, `import_declaration` | `import_statement`, `import_from_statement` | `preproc_include` |

**SQLite 表結構調整：**
```sql
CREATE TABLE module_apis (
  id INTEGER PRIMARY KEY,
  module_name TEXT,
  language TEXT,          -- 🌟 你提到的：'ts', 'python', 'c', 'vue'
  universal_kind TEXT,    -- 🌟 抽象化後的通用概念：'Function', 'Class', 'Interface'
  raw_ast_kind TEXT,      -- 保留原始 AST 節點名稱（除錯用）：'function_declaration'
  api_name TEXT,          -- 函數/類別名稱
  signature TEXT,         -- 統一格式化後的簽名（例如：將 C 的指標和 TS 的泛型統一轉為文字描述）
  file_path TEXT
);
```
**💡 關鍵思維**：AI Agent 在查詢時，**只應該查詢 `universal_kind`**（例如：`SELECT * WHERE universal_kind = 'Function'`），而永遠不應該去碰 `raw_ast_kind`。這層抽象化是保護 AI 不崩潰的防火牆。

### 2. IoT 特有思維：從「實作邊界」升級到「通訊合約邊界 (Communication Contract)」

在純 Web 專案中，模組的邊界是「函數呼叫」。但在 IoT 專案中，前後端與嵌入式設備之間的邊界，其實是 **「通訊協議與資料格式」**。

AI 在組合 IoT 專案時，最容易出錯的地方不是「函數寫錯」，而是 **「C 端發布的 MQTT Payload，跟 Node 端訂閱解析的格式對不上」**。

**補充方案：在 DB 中建立「通訊合約表」**

你需要利用 Tree-sitter 或正則表達式，額外提取出跨語言的「通訊節點」：

```sql
CREATE TABLE communication_contracts (
  id INTEGER PRIMARY KEY,
  protocol TEXT,          -- 'MQTT', 'HTTP', 'WebSocket', 'gRPC', 'UART'
  topic_or_path TEXT,     -- 例如 MQTT 的 'sensor/temperature/#' 或 HTTP 的 '/api/v1/data'
  payload_schema TEXT,    -- 🌟 關鍵：資料結構的 JSON Schema 或 TypeScript Interface 定義
  provider_lang TEXT,     -- 誰負責發送/提供？例如 'c' (嵌入式)
  consumer_lang TEXT,     -- 誰負責接收/消費？例如 'ts' (後端)
  description TEXT
);
```

**AI Agent 的視角：**
當 AI 要開發「後端接收溫度數據」的功能時，它不僅會查 `module_apis`，還會查 `communication_contracts`。
AI 會看到：「喔，C 端透過 MQTT 發送 `sensor/temp`，Payload 是 `{ "val": number, "ts": number }`」。於是 AI 在寫 Node.js 後端時，會**自動生成完全匹配的 TypeScript Interface 和解析邏輯**，徹底消除跨語言的整合 Bug。

### 3. 關於 `ast-grep` 的定位與使用建議

你提到利用 `ast-grep` 的多語言統一規則，這是一個很好的思路，但需要釐清它的最佳使用場景：

*   **Tree-sitter (底層解析)**：適合用來做 **「大規模掃描與提取」**。因為它極快，且能產出完整的 AST。你的「萃取寫入 DB」流程，建議還是以 Tree-sitter 為主，搭配你寫的 Adapter 層。
*   **ast-grep (高階搜尋與重構)**：它的強項在於 **「基於模式的匹配與修改」**。它的 YAML 規則語法（如 `$FUNC`, `$CLASS`）本身就是你提到的「多語言統一抽象」！

**💡 實作建議：兩者結合**
1.  **提取階段**：用 Tree-sitter 掃描，轉成你的 `Universal Core Concepts` 存入 SQLite。
2.  **AI 修改/生成階段**：當 AI Agent 需要去「修改」現有程式碼（例如：在 Go 和 TS 的 Router 中同時加入一個新路由）時，讓 AI 生成 **`ast-grep` 的 YAML 規則**。
    *   因為 `ast-grep` 的 `$FUNC` 規則可以同時適用於 TS 和 Go，AI 只需要寫一次邏輯，就能跨語言精準修改程式碼，不用管底層 AST 的差異。

### 4. 嵌入式端 (C/C++) 的特殊挑戰：資源與硬體標記

C/C++ 與 TS/Python 最大的不同，在於它與「硬體資源」綁定。AI 在生成 C 語言程式碼時，如果不知道硬體限制，會寫出記憶體爆炸或腳位衝突的廢碼。

**補充方案：硬體上下文 (Hardware Context)**
在掃描 C/C++ 程式碼時，特別提取以下資訊存入 DB：
*   **Memory Constraints**：例如 `#define MAX_BUFFER_SIZE 1024`。
*   **Pin/Hardware Mapping**：例如 `#define DHT11_PIN 4`。
*   **RTOS Tasks/Threads**：如果使用了 FreeRTOS，提取 `xTaskCreate` 的節點，記錄任務名稱與優先級。

讓 AI 在生成 C 語言模組時，能自動參考這些硬體限制，避免配置過大的 Buffer 或重複使用 GPIO。

### ❌ 為什麼不建議「整合成一筆資料」？

如果你把 `login` 設計成一筆資料，裡面同時包含後端 Controller 程式碼和前端 `LoginPage.vue` 程式碼：

1. **Token 嚴重浪費（最致命）**：
   如果今天甲方說：「幫我把後端登入的密碼加密演算法從 bcrypt 換成 argon2」。

2. **耦合過高，更新災難**：
   前後端的變更頻率通常不同。如果前端只是把「登入按鈕從藍色改成紅色」，後端 API 完全沒變，你卻必須把這「一整筆」包含後端邏輯的資料重新更新到資料庫，這在工程上是非常不優雅的。
3. **不符合 AI 的「按需載入 (On-Demand)」原則**：
   AI 應該像查字典一樣，需要哪一頁就翻哪一頁，而不是被迫吞下整本書。

### ✅ 正確做法：分開列舉 + 建立「功能映射 (Feature Mapping)」

我們應該在資料庫中將前後端視為**獨立的實體**，但用一張 **「關聯表」** 把它們綁在同一個「業務功能」下。

#### 1. 後端 API 表 (Backend APIs) - 獨立存放
專注於後端的邊界、依賴與實作。
```sql
CREATE TABLE backend_apis (
  id INTEGER PRIMARY KEY,
  module_name TEXT,       -- 'auth'
  api_name TEXT,          -- 'login'
  http_method TEXT,       -- 'POST'
  endpoint TEXT,          -- '/api/v1/auth/login'
  signature TEXT,         -- 後端函數簽名或 Controller 方法
  language TEXT,          -- 'ts', 'go', 'python'

```

#### 2. 前端元件/邏輯表 (Frontend UI/Logic) - 獨立存放
專注於前端的元件、狀態管理與 UI 邏輯。
```sql
CREATE TABLE frontend_components (
  id INTEGER PRIMARY KEY,
  module_name TEXT,       -- 'auth'
  component_name TEXT,    -- 'LoginPage', 'useLoginHook'
  component_type TEXT,    -- 'page', 'component', 'hook', 'store'
  language TEXT,          -- 'vue', 'react' (或 'ts')
  props_signature TEXT,   -- 前端 Props 定義
  -- ... 其他前端專屬欄位 (UI 狀態、路由路徑等)
);
```

#### 3. 🌟 核心殺手鐧：功能映射與合約表 (Feature Mapping & Contracts)
這張表是連接前後端的橋樑，也是 AI 進行「全端開發」時的導航地圖。
```sql
CREATE TABLE feature_contracts (
  id INTEGER PRIMARY KEY,
  feature_name TEXT,      -- '使用者登入功能' (業務層級的名稱)
  backend_api_id INTEGER, -- 關聯到 backend_apis 的 id
  frontend_comp_id INTEGER, -- 關聯到 frontend_components 的 id
  
  -- 🌟 關鍵：前後端溝通的「合約」 (API Payload / DTO)
  request_schema TEXT,    -- 前端傳給後端的 JSON 結構 (例如: { username: string, password: string })
  response_schema TEXT,   -- 後端回傳給前端的 JSON 結構 (例如: { token: string, user: UserDto })
  
  description TEXT        -- 這個功能的業務邏輯描述
);
```

#### 情境 A：只修改後端邏輯（例如：修改 JWT 過期時間）
* **AI 的動作**：只查詢 `backend_apis` 表，讀取 `login` 的後端實作。
* **Token 消耗**：極低。前端程式碼完全不被載入。

* **Token 消耗**：極低。後端邏輯完全不被載入。

#### 情境 C：從零開發「使用者登入功能」（全端開發）

### 🌟 為什麼「雙 DB 模式」是神來之筆？（4 大核心優勢）

#### 1. 絕對的「唯讀保護」，防止 AI 幻覺破壞核心
*   **痛點**：如果所有程式碼都在同一個 DB，當 AI 在開發甲方案的「訂單模組」時，產生了幻覺，可能會「自作聰明」地去修改你寫好的 `auth` 底層邏輯，導致整個系統崩潰。
*   **雙 DB 解法**：將 `global_modules.db` 設定為 **Read-Only（唯讀）**。AI Agent 在 MCP Server 層級被嚴格限制，**只能查詢 Global DB，絕對無法寫入或修改**。這從物理層面杜絕了 AI 破壞基礎建設的可能。

#### 2. 極致的 Context 隔離與 Token 節省
*   **痛點**：AI 的 Context Window 是有限的。如果混在一起，AI 每次寫業務邏輯，都要被迫載入一堆它根本不需要改的公共模組細節。
*   **雙 DB 解法**：
    *   當 AI 任務是「開發甲方案的溫室監控業務」時，它**只挂载 `project_local.db`**，Context 極度純粹。
    *   只有當 AI 需要「呼叫」底層功能時，才去 `global_modules.db` 查詢 API 邊界（只查簽名，不查實作）。

#### 3. 完美的資產沉澱與版本控制
*   `global_modules.db`：跟隨你的「公共 IoT 框架」版本控制。框架升級（例如 MQTT 核心優化了重連機制），只需更新這一個 DB，所有歷史專案的 AI 都能立刻學到新知識。
*   `project_local.db`：跟隨專案的 `.git`。它記錄了這個專案獨有的「業務知識」。專案交付後，這個 DB 就是該專案最寶貴的「機器可讀交接文件」。

#### 4. SQLite 原生的跨庫查詢 (ATTACH DATABASE)
SQLite 原生支援在同一個連線中掛載多個資料庫，這讓 AI Agent 寫 SQL 查詢變得極度優雅：
```sql
-- AI Agent 在 MCP Server 中執行的 SQL
ATTACH '/path/to/global_modules.db' AS global;
ATTACH '/path/to/project_local.db' AS local;

-- AI 可以輕鬆進行跨庫關聯查詢：
-- 例如：找出專案特有的 API，以及它們依賴的公共模組 API
SELECT local.api_name, local.module_name, global.signature 
FROM local.module_apis AS local
JOIN global.module_dependencies AS dep ON local.module_name = dep.source_module
JOIN global.module_apis AS global ON dep.target_module = global.module_name;
```

### 💡 進階補充方案：讓雙 DB 更強大的 3 個思維

雖然雙 DB 已經很棒，但在真實的 IoT 專案開發中，還會遇到一些邊角情況。以下補充 3 個進階機制：

#### 補充一：引入「覆寫與擴展機制 (Override & Extension)」
**情境**：公共的 `auth` 模組預設使用 JWT，但甲方案甲方硬性規定必須使用他們內部的 SSO 系統。AI 該怎麼處理？
**解法**：在 `project_local.db` 中設計一個 `module_overrides` 表。
```sql
CREATE TABLE module_overrides (
  global_module_name TEXT,  -- 'auth'
  override_type TEXT,       -- 'replace', 'extend', 'hook'
  local_impl_path TEXT,     -- 指向專案特有的實作檔案
  description TEXT          -- 「甲方案使用內部 SSO 替換標準 JWT 簽發邏輯」
);
```
**AI 視角**：AI 在組合專案時，會先查 `global`，然後**檢查 `local` 有沒有覆寫**。如果有，AI 就會優先使用甲方案的 SSO 邏輯，而不是盲目使用公共模組。

#### 補充二：AI 查詢的「降級策略 (Fallback Strategy)」
在設計 MCP Server 給 AI 用的 Tool 時，可以內建查詢優先級：
1.  **優先查 Local**：`SELECT * FROM local.module_apis WHERE name = 'getTemperature'`
2.  **若無，再查 Global**：`SELECT * FROM global.module_apis WHERE name = 'getTemperature'`
這確保了 AI 永遠優先使用「專案特有/客製化」的邏輯，如果沒有，才去使用「公共標準」邏輯。這完全符合軟體開發的直覺。

#### 補充三：硬體上下文 (Hardware Context) 的雙層設計
在 IoT 專案中，硬體限制也是分層的：
*   **Global 層**：記錄「通用開發板」的預設限制（例如 ESP32 的 GPIO 腳位定義、最大記憶體）。
*   **Local 層**：記錄「甲方案特定硬體」的客製化配置（例如甲方案把 DHT11 接在 Pin 4，且限制了 Buffer 大小為 512）。
AI 在生成 C/C++ 嵌入式程式碼時，必須**以 Local 的硬體上下文為準，Global 作為預設值 (Fallback)**。

## 二、解決方案：三種同步策略的取捨

### 策略 A：全量重建（Full Rebuild）— 最簡單、最正確

**原理**：每次觸發時，**清空 DB，重新掃描整個專案**。

```bash
# 觸發時機：Git commit 後、CI/CD 流程中
npm run sync-graph -- --full-rebuild
```

**腳本邏輯**：
```javascript
// 1. 刪除所有舊資料
db.exec('DELETE FROM module_apis');
db.exec('DELETE FROM module_dependencies');

// 2. 重新掃描所有檔案
for (const file of getAllSourceFiles()) {
  const ast = treeSitter.parse(file);
  extractAndInsert(ast);
}
```

**✅ 優點**：
- 邏輯最簡單，**100% 不會有髒資料**。
- 完美處理刪除、重構、分支切換。

**❌ 缺點**：
- 大型專案（幾千個檔案）掃描耗時（可能 10-30 秒）。
- 不適合「打字時即時同步」。

**💡 適用場景**：Git commit 後、CI/CD 流程、專案初始化。

---

### 策略 B：增量同步（Incremental Sync）— 快速、但複雜

**原理**：只處理「變更的檔案」，透過比對檔案的 `mtime`（修改時間）或 `git diff` 來決定要重新解析哪些檔案。

**腳本邏輯**：
```javascript
// 1. 取得自上次同步以來變更的檔案
const changedFiles = getChangedFilesSinceLastSync();

// 2. 針對每個變更檔案
for (const file of changedFiles) {
  if (isDeleted(file)) {
    // 刪除該檔案在 DB 中的所有記錄
    db.exec(`DELETE FROM module_apis WHERE file_path = ?`, [file]);
  } else {
    // 先刪除舊記錄，再重新解析插入新記錄
    db.exec(`DELETE FROM module_apis WHERE file_path = ?`, [file]);
    const ast = treeSitter.parse(file);
    extractAndInsert(ast);
  }
}
```

**✅ 優點**：
- 速度極快（只處理變更的檔案，毫秒級）。
- 適合 File Watcher 即時監聽。

**❌ 缺點**：
- 邏輯複雜，要處理「刪除」、「移動」、「重構」等多種邊角情況。
- 如果中途崩潰，DB 可能處於不一致狀態。

**💡 適用場景**：開發過程中的即時同步（搭配 File Watcher）。

---

### 策略 C：混合策略（Hybrid）— 🌟 推薦的生產級做法

結合 A 和 B 的優點：

1. **開發時（即時）**：用 File Watcher 觸發**增量同步**（策略 B），讓 AI 能立刻看到最新變更。
2. **Git Commit 後（定期）**：觸發**全量重建**（策略 A），確保 DB 絕對乾淨。
3. **分支切換時**：強制觸發**全量重建**（因為 `git checkout` 可能一次變更幾百個檔案）。

```javascript

  fullRebuild();
} else {
  incrementalSync();
}
```

## 四、進階：處理「重構」的終極挑戰

重構（例如把 `getUser()` 改名為 `findUserById()`）是最難處理的場景。單純的增量同步可能會留下「幽靈記錄」。

**解法：引入「指紋（Fingerprint）機制」**

在 DB 中為每個 API 計算一個「指紋」（例如：檔案路徑 + 行號 + 函數名稱的 Hash）。

```sql
CREATE TABLE module_apis (
  id INTEGER PRIMARY KEY,
  file_path TEXT,
  line_start INTEGER,
  api_name TEXT,
  fingerprint TEXT,  -- 🌟 關鍵：hash(file_path + line_start + api_name)
  -- ...
);
```

**同步邏輯**：
1. 掃描檔案，計算每個 API 的指紋。
2. 比對 DB 中的指紋：
   - **新指紋**：插入新記錄。
   - **消失的指紋**：刪除舊記錄。
   - **相同的指紋**：保留（可選擇性更新內容）。

這樣即使函數改名，因為「行號 + 檔案路徑」變了，指紋也會變，系統會正確地「刪除舊的、插入新的」，不會有幽靈資料。

### 一、 為什麼 LLM 寫複雜 SQL 一定會出錯？

LLM 的本質是「機率性的文字接龍」，而不是「嚴謹的邏輯編譯器」。當它面對複雜 SQL 時，會遇到以下三個物理限制：

1. **Context Window 過載 (Lost in the Middle)**：
   如果你的 DB 有 20 張表，每張表有 15 個欄位。當你把整個 Schema 塞給 AI 時，它的注意力會被分散。它可能會把 `global_modules` 表的 `module_id` 和 `project_local` 表的 `api_id` 搞混。
2. **命名風格不一致的猜測**：
   人類開發者有時會混用 `user_id`、`userId`、`author_id`。LLM 沒有編譯器幫它檢查，它只能「猜」。一旦猜錯，SQL 就會報 `no such column` 錯誤。
3. **多表 JOIN 的邏輯迷宮**：
   三張表以上的 JOIN，需要精準記住 A 表的哪個外鍵連到 B 表的主鍵。LLM 很容易在文字生成時，憑「感覺」把 `A.id = B.a_id` 寫成 `A.id = B.id`。

**結論：如果你直接給 AI 一個 `execute_sql(query: string)` 的工具，讓它自己拼 SQL，你的系統在生產環境中絕對會頻繁崩潰。**

### 二、 MCP 是可靠的解決作法嗎？

**答案是：MCP 本身只是協議，但「透過 MCP 實作特定的設計模式」，是解決這個問題最可靠、也是業界標準的做法。**

核心思維轉換：**「絕對不要讓 LLM 直接寫 SQL，讓 LLM 寫『意圖』，由 MCP Server 底層的確定性程式碼來寫 SQL。」**

以下是利用 MCP 解決 SQL 幻覺的 **3 個實戰策略**：

#### 策略 1：降維打擊 —— 封裝「高階語意工具 (Semantic Tools)」 (🌟 最強烈推薦)

**錯誤的 MCP 設計（暴露底層）：**
```typescript
// ❌ 讓 AI 自己寫 SQL，必然出錯
@mcpTool("execute_sql")
function executeSql(query: string) {
  return db.exec(query); 
}
// AI 呼叫：execute_sql("SELECT * FROM module_apis a JOIN global_modules g ON a.id = g.module_id") -> 💥 欄位寫錯
```

**正確的 MCP 設計（封裝業務邏輯）：**
在 MCP Server 端，由**人類工程師**把複雜的 JOIN 寫死在 TypeScript 程式碼裡，對外只暴露簡單的「業務參數」。

```typescript
// ✅ AI 只需要懂業務參數，底層 SQL 由人類寫死，100% 正確
@mcpTool("get_api_with_dependencies")
function getApiWithDependencies(moduleName: string, apiName: string) {
  // 底層由你寫好絕對正確的 SQL
  const sql = `
    SELECT a.api_name, a.signature, g.module_name as global_dep 
    FROM local.module_apis a
    LEFT JOIN global.module_dependencies d ON a.module_name = d.source_module
    LEFT JOIN global.module_apis g ON d.target_module = g.module_name
    WHERE a.module_name = ? AND a.api_name = ?;
  `;
  return db.prepare(sql).all(moduleName, apiName);
}
```
**AI 的視角**：AI 只需要呼叫 `get_api_with_dependencies("auth", "login")`。它完全不需要知道有幾張表、JOIN 怎麼寫，徹底消滅了 SQL 幻覺。

#### 策略 2：精準餵食 —— 動態 Schema 注入 (Dynamic Schema Injection)

如果你有些「即興查詢」的需求，必須讓 AI 寫 SQL，那麼 MCP 必須在 AI 寫 SQL **之前**，先幫它「縮小範圍」。

不要一次性把整個 DB 的 Schema 給 AI。實作一個 MCP 工具：

```typescript
@mcpTool("get_table_schema")
function getTableSchema(tableNames: string[]) {
  // 只回傳 AI 當前需要的表的欄位定義
  return db.prepare(`SELECT sql FROM sqlite_master WHERE name IN (?)`).all(tableNames);
}
```
**工作流**：
1. AI 決定要查 `auth` 的依賴。
2. AI 先呼叫 `get_table_schema(["module_apis", "module_dependencies"])`。
3. MCP 回傳這兩張表的精確 `CREATE TABLE` 語句。
4. AI 看著**只有這兩張表**的精確 Schema，再寫出 JOIN 語句。
*這能將 LLM 的注意力集中，大幅降低寫錯欄位的機率。*

#### 策略 3：防呆機制 —— 執行前驗證與自動修復 (Self-Correction)

即使做了前兩步，AI 還是可能寫錯。MCP Server 可以充當「編譯器」，在 SQL 真正執行前進行攔截。

```typescript
@mcpTool("execute_sql_with_validation")
function executeSqlWithValidation(query: string) {
  try {
    // 1. 嘗試準備 SQL 語句 (SQLite 會檢查欄位是否存在)
    const stmt = db.prepare(query);
    return stmt.all();
  } catch (error) {
    // 2. 如果報錯 (例如 no such column: user_id)
    // 3. 捕獲錯誤，並把錯誤訊息連同正確的 Schema 餵回給 AI
    return {
      error: true,
      message: `SQL 執行失敗: ${error.message}`,
      hint: "請檢查欄位名稱，並使用 get_table_schema 工具確認正確的欄位。"
    };
  }
}
```
**AI 的視角**：AI 寫錯 SQL -> MCP 回傳錯誤與提示 -> AI 讀取錯誤 -> AI 自動修正 SQL 並再次呼叫。這形成了一個**自動除錯閉環**，不需要人類介入。

---

### 三、 總結：MCP 在你的架構中的真正定位

回到你的「雙 DB (Global + Local)」架構，MCP 不僅僅是一個「讓 AI 讀取資料庫的通道」，它更是 **「保護資料庫不被 AI 幻覺破壞的防火牆」**。

一個成熟的、生產級的 Code Graph MCP Server 應該這樣設計：

1. **90% 的場景**：使用**高階語意工具**（如 `search_api`, `get_dependencies`, `check_override`）。AI 傳遞字串參數，MCP 底層執行寫死的、經過測試的複雜 SQL。這保證了**絕對的穩定性與正確性**。
2. **10% 的場景**（例如開發者想做一些即興的資料分析）：開放**受控的 SQL 工具**，搭配動態 Schema 注入與執行前驗證，讓 AI 有自我修正的機會。

**你的擔憂非常專業。** 意識到「LLM 會寫錯 SQL」並尋求架構上的解法，正是從「AI 玩具開發者」進化為「AI 系統架構師」的關鍵轉折。用 MCP 把複雜的 SQL 封裝在底層，讓 AI 專注於理解業務意圖，這才是真正可靠的工程實踐！

### 一、 核心基礎表 (Global & Local 共用)
*這些表存在於 `global_modules.db` 和 `project_local.db` 中，結構完全一致。*

#### 1. 模組宏觀資訊表 (Modules)
記錄模組級別的高階摘要、依賴與注意事項。
```sql
CREATE TABLE modules (
  module_name TEXT PRIMARY KEY,       -- 模組名稱 (e.g., 'auth', 'mqtt_core')
  language TEXT NOT NULL,             -- 主要語言 (e.g., 'ts', 'go', 'c', 'vue')
  description TEXT,                   -- 宏觀業務描述 (供 AI 理解模組用途)
  dependencies TEXT,                  -- JSON 陣列: 依賴的其他模組 (e.g., '["config", "database"]')
  gotchas TEXT,                       -- 隱藏的坑/注意事項 (e.g., '必須先呼叫 init()')
  tags TEXT,                          -- JSON 陣列: 搜尋標籤 (e.g., '["金流", "付款", "stripe"]')
  fingerprint TEXT NOT NULL,          -- 指紋: hash(description + dependencies + gotchas)，用於增量同步判斷是否變更
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

#### 2. 後端 API 邊界表 (Backend APIs)
**核心設計：每個 API 獨立一筆資料（扁平化 1:N）**，極致節省 Token。
```sql
CREATE TABLE backend_apis (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  module_name TEXT NOT NULL,          -- 關聯 modules 表
  language TEXT NOT NULL,             -- 'ts', 'go', 'python'
  
  -- 🌟 跨語言抽象化層
  universal_kind TEXT NOT NULL,       -- 統一概念: 'Function', 'Class', 'Interface', 'Controller'
  raw_ast_kind TEXT,                  -- 原始 AST 節點 (e.g., 'function_declaration', 'function_definition')
  
  api_name TEXT NOT NULL,             -- API/函數/類別名稱 (e.g., 'login', 'UserController')
  http_method TEXT,                   -- HTTP 方法 (GET, POST...)，若非 HTTP 則為 NULL
  endpoint TEXT,                      -- 路由路徑 (e.g., '/api/v1/auth/login')
  signature TEXT,                     -- 完整的型別簽名 (e.g., 'login(dto: LoginDto): Promise<Token>')
  
  description TEXT,                   -- 該 API 的詳細描述
  example_code TEXT,                  -- 黃金範例程式碼 (Few-shot prompting 用)
  
  file_path TEXT NOT NULL,            -- 原始碼檔案路徑
  line_start INTEGER,                 -- 起始行號
  line_end INTEGER,                   -- 結束行號
  fingerprint TEXT NOT NULL,          -- 指紋: hash(file_path + line_start + api_name + signature)
  
  UNIQUE(module_name, api_name)
);
```

#### 3. 前端元件/邏輯表 (Frontend Components)
與後端物理分離，專注於 UI 與狀態管理。
```sql
CREATE TABLE frontend_components (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  module_name TEXT NOT NULL,
  language TEXT NOT NULL,             -- 'vue', 'react', 'ts'
  
  component_name TEXT NOT NULL,       -- 元件/Hook/Store 名稱 (e.g., 'LoginPage', 'useAuthStore')
  component_type TEXT NOT NULL,       -- 'page', 'component', 'hook', 'store', 'utils'
  
  props_signature TEXT,               -- Props / 參數型別定義
  description TEXT,
  example_code TEXT,
  
  file_path TEXT NOT NULL,
  line_start INTEGER,
  line_end INTEGER,
  fingerprint TEXT NOT NULL,
  
  UNIQUE(module_name, component_name)
);
```

#### 4. 模組依賴關係表 (Dependency Graph - 邊表)
構建知識圖譜的核心，記錄模組間的呼叫與依賴。
```sql
CREATE TABLE module_dependencies (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  source_module TEXT NOT NULL,        -- 呼叫方/依賴方 (e.g., 'order')
  target_module TEXT NOT NULL,        -- 被呼叫方/被依賴方 (e.g., 'auth')
  import_path TEXT,                   -- 具體的 import 路徑
  dependency_type TEXT,               -- 'import', 'extends', 'calls', 'uses'
  
  UNIQUE(source_module, target_module, import_path)
);
```

### 二、 專案特有表 (Project Local 專屬)
*這些表主要存在於 `project_local.db` 中，用於處理前後端關聯、IoT 通訊與硬體限制。Global DB 可選擇性保留空表或結構。*

#### 5. 前後端功能合約表 (Feature Contracts)
**核心設計：邏輯關聯**，確保 AI 在生成全端程式碼時，前後端 Payload 一致。
```sql
CREATE TABLE feature_contracts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  feature_name TEXT NOT NULL,         -- 業務功能名稱 (e.g., '使用者登入功能')
  
  -- 關聯到後端與前端實體
  backend_api_id INTEGER REFERENCES backend_apis(id),
  frontend_comp_id INTEGER REFERENCES frontend_components(id),
  
  -- 🌟 跨端溝通的絕對合約
  request_schema TEXT,                -- 前端傳後端的 JSON Schema 或 TS Interface 定義
  response_schema TEXT,               -- 後端回傳前端的 JSON Schema 或 TS Interface 定義
  
  description TEXT,
  UNIQUE(feature_name)
);
```

#### 6. IoT 通訊合約表 (Communication Contracts)
解決嵌入式 (C/C++) 與後端 (Node/Go) 跨語言整合的痛點。
```sql
CREATE TABLE communication_contracts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  protocol TEXT NOT NULL,             -- 'MQTT', 'HTTP', 'WebSocket', 'gRPC', 'UART'
  topic_or_path TEXT NOT NULL,        -- MQTT Topic (e.g., 'sensor/temp/#') 或 HTTP Path
  
  payload_schema TEXT NOT NULL,       -- 🌟 關鍵：Payload 的結構定義 (JSON Schema / C Struct / TS Interface)
  
  provider_lang TEXT,                 -- 資料提供方語言 (e.g., 'c' 嵌入式)
  consumer_lang TEXT,                 -- 資料消費方語言 (e.g., 'ts' 後端)
  
  description TEXT,
  UNIQUE(protocol, topic_or_path)
);
```

#### 7. 嵌入式硬體上下文表 (Hardware Context)
防止 AI 在寫 C/C++ 時產生記憶體爆炸或腳位衝突的幻覺。
```sql
CREATE TABLE hardware_context (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  module_name TEXT NOT NULL,          -- 關聯的模組 (e.g., 'dht11_driver')
  target_board TEXT,                  -- 目標開發板 (e.g., 'ESP32', 'STM32', 'RaspberryPi')
  
  memory_limit_bytes INTEGER,         -- 記憶體限制
  pin_mapping TEXT,                   -- JSON: 腳位配置 (e.g., '{"data_pin": 4, "vcc_pin": "3V3"}')
  rtos_tasks TEXT,                    -- JSON: FreeRTOS 任務配置 (e.g., '[{"name": "sensor_task", "priority": 2}]')
  
  description TEXT
);
```

#### 8. 模組覆寫表 (Module Overrides - Local 專屬)
**雙 DB 架構的核心**：允許 Local 專案替換或擴展 Global 的公共模組。
```sql
CREATE TABLE module_overrides (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  global_module_name TEXT NOT NULL,   -- 被覆寫的 Global 模組 (e.g., 'auth')
  override_type TEXT NOT NULL,        -- 'replace' (完全替換), 'extend' (繼承擴展), 'hook' (攔截)
  
  local_impl_path TEXT NOT NULL,      -- 指向 Local 專案中的實作檔案路徑
  description TEXT,                   -- 為什麼要覆寫 (e.g., '甲方案需使用內部 SSO 替換 JWT')
  
  UNIQUE(global_module_name)
);
```

### 三、 FTS5 全文檢索虛擬表 (AI 精準搜尋引擎)
*為了讓 AI 能夠毫秒級模糊搜尋，且節省 Token，必須為核心文字欄位建立 FTS5 索引。*

```sql
-- 1. 後端 API 搜尋 (AI 找函數、找簽名)
CREATE VIRTUAL TABLE backend_apis_fts USING fts5(
  module_name, 
  api_name, 
  signature, 
  description,
  tags, -- 來自 modules 表的 tags，可透過 trigger 同步或冗余存儲
  content='backend_apis', 
  content_rowid='id'
);

-- 2. 前端元件搜尋
CREATE VIRTUAL TABLE frontend_components_fts USING fts5(
  module_name, 
  component_name, 
  props_signature, 
  description,
  content='frontend_components', 
  content_rowid='id'
);

-- 3. 功能合約與通訊合約搜尋 (AI 找業務邏輯、找 MQTT Topic)
CREATE VIRTUAL TABLE contracts_fts USING fts5(
  feature_name,
  protocol,
  topic_or_path,
  payload_schema,
  description,
  content='feature_contracts', -- 註：實際實作時可將 feature_contracts 與 communication_contracts 合併查詢，或建兩個 FTS 表
  content_rowid='id'
);
```

有了這份完美的 Schema，你的 MCP Server **絕對不要**暴露 `execute_sql`，而是應該封裝以下高階工具，讓 AI 乖乖傳參數，徹底消滅 SQL 幻覺：

#### 🛠️ 工具 1：`search_code_entity` (兩階段檢索)

#### 🛠️ 工具 2：`get_entity_details` (按需載入)

#### 🛠️ 工具 3：`get_module_dependencies` (圖譜遍歷)
```typescript
// AI 呼叫：get_module_dependencies({ module_name: "order", depth: 2 })
```

#### 🛠️ 工具 4：`check_module_override` (雙 DB 防呆)

```typescript
// AI 呼叫：check_module_override({ global_module: "auth" })
```

#### 🛠️ 工具 5：`get_iot_contract` (跨語言整合)


### Step 1.2：建立跨語言 AST 適配器

**檔案：`scripts/adapters/typescript-adapter.ts`**
```typescript
import Parser from 'web-tree-sitter';

export interface ExtractedAPI {
  universal_kind: 'Function' | 'Class' | 'Interface' | 'Controller';
  raw_ast_kind: string;
  api_name: string;
  signature: string;
  description: string;
  file_path: string;
  line_start: number;
  line_end: number;
  imports: string[];
}

export class TypeScriptAdapter {
  private parser: Parser;
  
  async init() {
    await Parser.init();
    this.parser = new Parser();
    const Lang = await Parser.Language.load(
      'node_modules/tree-sitter-wasms/out/tree-sitter-typescript.wasm'
    );
    this.parser.setLanguage(Lang);
  }
  
  extractAPIs(filePath: string, sourceCode: string): ExtractedAPI[] {
    const tree = this.parser.parse(sourceCode);
    const apis: ExtractedAPI[] = [];
    
    // 走訪 AST，找出所有 function_declaration
    const cursor = tree.walk();
    this.traverse(cursor, (node) => {
      if (node.type === 'function_declaration') {
        const nameNode = node.childForFieldName('name');
        const paramsNode = node.childForFieldName('parameters');
        
        apis.push({
          universal_kind: 'Function',
          raw_ast_kind: 'function_declaration',
          api_name: nameNode?.text || 'anonymous',
          signature: `(${paramsNode?.text || ''})`,
          description: this.extractJSDoc(node),
          file_path: filePath,
          line_start: node.startPosition.row,
          line_end: node.endPosition.row,
          imports: this.extractImports(tree.rootNode)
        });
      }
      
      if (node.type === 'class_declaration') {
        // 類似邏輯處理 class...
      }
    });
    
    return apis;
  }
  
  private extractJSDoc(node: Parser.SyntaxNode): string {
    // 向上找 JSDoc 註解
    const prev = node.previousSibling;
    if (prev?.type === 'comment' && prev.text.startsWith('/**')) {
      return prev.text;
    }
    return '';
  }
  
  private extractImports(rootNode: Parser.SyntaxNode): string[] {
    const imports: string[] = [];
    rootNode.children
      .filter(n => n.type === 'import_statement')
      .forEach(n => imports.push(n.text));
    return imports;
  }
  
  private traverse(cursor: Parser.TreeCursor, callback: (node: Parser.SyntaxNode) => void) {
    callback(cursor.currentNode);
    if (cursor.gotoFirstChild()) {
      do {
        this.traverse(cursor, callback);
      } while (cursor.gotoNextSibling());
      cursor.gotoParent();
    }
  }
}
```

### Step 1.3：建立全量重建腳本

**檔案：`scripts/build-graph.ts`**
```typescript
#!/usr/bin/env ts-node
import { initDatabase } from './init-db';
import { TypeScriptAdapter } from './adapters/typescript-adapter';
import * as fs from 'fs';
import * as path from 'path';
import * as crypto from 'crypto';

async function main() {
  const mode = process.argv[2] || 'full'; // 'full' | 'incremental'
  
  // 初始化雙 DB
  const globalDb = initDatabase('.codegraph/global_modules.db');
  const localDb = initDatabase('.codegraph/project_local.db');
  
  const tsAdapter = new TypeScriptAdapter();
  await tsAdapter.init();
  
  if (mode === 'full') {
    console.log('🔄 執行全量重建...');
    globalDb.exec('DELETE FROM backend_apis; DELETE FROM modules;');
    localDb.exec('DELETE FROM backend_apis; DELETE FROM modules;');
  }
  
  // 掃描 global-modules 目錄
  await scanDirectory('global-modules', globalDb, tsAdapter);
  
  // 掃描 src 目錄
  await scanDirectory('src', localDb, tsAdapter);
  
  console.log('✅ 知識圖譜構建完成！');
}

async function scanDirectory(
  dir: string, 
  db: Database.Database, 
  adapter: TypeScriptAdapter
) {
  const files = getAllFiles(dir);
  
  for (const file of files) {
    const content = fs.readFileSync(file, 'utf-8');
    const apis = adapter.extractAPIs(file, content);
    
    for (const api of apis) {
      const fingerprint = crypto
        .createHash('md5')
        .update(`${file}:${api.line_start}:${api.api_name}:${api.signature}`)
        .digest('hex');
      
      const moduleName = extractModuleName(file);
      
      db.prepare(`
        INSERT OR REPLACE INTO backend_apis 
        (module_name, language, universal_kind, raw_ast_kind, api_name, 
         signature, description, file_path, line_start, line_end, fingerprint)
        VALUES (?, 'ts', ?, ?, ?, ?, ?, ?, ?, ?, ?)
      `).run(
        moduleName, api.universal_kind, api.raw_ast_kind, api.api_name,
        api.signature, api.description, api.file_path, 
        api.line_start, api.line_end, fingerprint
      );
    }
  }
}

main().catch(console.error);
```

```typescript
import { Server } from '@modelcontextprotocol/sdk/server/index.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';
import { CallToolRequestSchema, ListToolsRequestSchema } from '@modelcontextprotocol/sdk/types.js';
import Database from 'better-sqlite3';

// 掛載雙 DB
const globalDb = new Database('.codegraph/global_modules.db', { readonly: true });
const localDb = new Database('.codegraph/project_local.db');

// 跨庫查詢設定
localDb.exec(`ATTACH '.codegraph/global_modules.db' AS global;`);

const server = new Server(
  { name: 'code-graph-mcp', version: '1.0.0' },
  { capabilities: { tools: {} } }
);

// 註冊工具列表
server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [
    {
      name: 'search_code_entity',
      description: '搜尋後端 API 或前端元件（使用關鍵字）',
      inputSchema: {
        type: 'object',
        properties: {
          keyword: { type: 'string' },
          entity_type: { 
            type: 'string', 
            enum: ['backend_api', 'frontend_component', 'all'] 
          },
          include_example: { type: 'boolean', default: false }
        },
        required: ['keyword']
      }
    },
    {
      name: 'get_module_dependencies',
      description: '查詢模組的依賴關係圖',
      inputSchema: {
        type: 'object',
        properties: {
          module_name: { type: 'string' },
          depth: { type: 'number', default: 1 }
        },
        required: ['module_name']
      }
    },
    {
      name: 'check_module_override',
      description: '檢查某個 Global 模組是否被 Local 專案覆寫',
      inputSchema: {
        type: 'object',
        properties: {
          global_module_name: { type: 'string' }
        },
        required: ['global_module_name']
      }
    },
    {
      name: 'get_iot_contract',
      description: '查詢 IoT 通訊合約（MQTT Topic、Payload 結構）',
      inputSchema: {
        type: 'object',
        properties: {
          topic: { type: 'string' }
        },
        required: ['topic']
      }
    },
    {
      name: 'get_feature_contract',
      description: '查詢前後端功能合約（Request/Response Schema）',
      inputSchema: {
        type: 'object',
        properties: {
          feature_name: { type: 'string' }
        },
        required: ['feature_name']
      }
    }
  ]
}));

// 實作工具邏輯
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  const { name, arguments: args } = request.params;
  
  switch (name) {
    case 'search_code_entity':
      return handleSearch(args);
    case 'get_module_dependencies':
      return handleDependencies(args);
    case 'check_module_override':
      return handleOverride(args);
    case 'get_iot_contract':
      return handleIoTContract(args);
    case 'get_feature_contract':
      return handleFeatureContract(args);
    default:
      throw new Error(`Unknown tool: ${name}`);
  }
});

// 🔑 核心：封裝複雜 SQL，AI 只需傳參數
function handleSearch(args: any) {
  const { keyword, entity_type = 'all', include_example = false } = args;
  
  let results: any[] = [];
  
  if (entity_type === 'backend_api' || entity_type === 'all') {
    // 優先查 Local，再查 Global
    const localResults = localDb.prepare(`
      SELECT id, module_name, api_name, signature, description${
        include_example ? ', example_code' : ''
      }
      FROM backend_apis_fts 
      WHERE backend_apis_fts MATCH ?
      LIMIT 20
    `).all(keyword);
    
    const globalResults = globalDb.prepare(`
      SELECT id, module_name, api_name, signature, description${
        include_example ? ', example_code' : ''
      }
      FROM backend_apis_fts 
      WHERE backend_apis_fts MATCH ?
      LIMIT 20
    `).all(keyword);
    
    results = [
      ...localResults.map(r => ({ ...r, source: 'local' })),
      ...globalResults.map(r => ({ ...r, source: 'global' }))
    ];
  }
  
  return {
    content: [{ 
      type: 'text', 
      text: JSON.stringify(results, null, 2) 
    }]
  };
}

function handleDependencies(args: any) {
  const { module_name, depth = 1 } = args;
  
  // 使用遞迴 SQL 查詢依賴圖
  const sql = `
    WITH RECURSIVE deps(module, target, depth) AS (
      SELECT source_module, target_module, 1 
      FROM module_dependencies 
      WHERE source_module = ?
      UNION ALL
      SELECT md.source_module, md.target_module, d.depth + 1
      FROM module_dependencies md
      JOIN deps d ON md.source_module = d.target
      WHERE d.depth < ?
    )
    SELECT DISTINCT * FROM deps;
  `;
  
  const results = localDb.prepare(sql).all(module_name, depth);
  
  return {
    content: [{ type: 'text', text: JSON.stringify(results, null, 2) }]
  };
}

function handleOverride(args: any) {
  const { global_module_name } = args;
  
  const override = localDb.prepare(`
    SELECT * FROM module_overrides WHERE global_module_name = ?
  `).get(global_module_name);
  
  if (override) {
    return {
      content: [{ 
        type: 'text', 
        text: `⚠️ 模組 ${global_module_name} 已被覆寫！\n` +
              `覆寫類型: ${override.override_type}\n` +
              `本地實作路徑: ${override.local_impl_path}\n` +
              `原因: ${override.description}` 
      }]
    };
  }
  
  return {
    content: [{ 
      type: 'text', 
      text: `✅ 模組 ${global_module_name} 使用 Global 預設實作` 
    }]
  };
}

function handleIoTContract(args: any) {
  const { topic } = args;
  
  const contract = localDb.prepare(`
    SELECT * FROM communication_contracts 
    WHERE topic_or_path LIKE ?
  `).all(`%${topic}%`);
  
  return {
    content: [{ type: 'text', text: JSON.stringify(contract, null, 2) }]
  };
}

function handleFeatureContract(args: any) {
  const { feature_name } = args;
  
  const contract = localDb.prepare(`
    SELECT fc.*, 
           ba.signature as backend_signature,
           fc_comp.props_signature as frontend_props
    FROM feature_contracts fc
    LEFT JOIN backend_apis ba ON fc.backend_api_id = ba.id
    LEFT JOIN frontend_components fc_comp ON fc.frontend_comp_id = fc_comp.id
    WHERE fc.feature_name LIKE ?
  `).all(`%${feature_name}%`);
  
  return {
    content: [{ type: 'text', text: JSON.stringify(contract, null, 2) }]
  };
}

// 啟動 MCP Server
async function run() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error('Code Graph MCP Server 已啟動');
}

run().catch(console.error);
```

## 🎯 落地檢查清單

按照以下順序實作，預計 **2-3 週** 可以完成 MVP：

### Week 1：基礎建設
- [ ] 初始化專案結構與安裝依賴
- [ ] 實作 `init-db.ts` 建立 Schema
- [ ] 實作 TypeScript AST 適配器
- [ ] 完成 `build-graph.ts` 全量重建
- [ ] 手動測試：掃描你的 `auth` 模組，確認資料正確寫入 DB

### Week 2：MCP Server
- [ ] 實作 MCP Server 主程式
- [ ] 實作 5 個核心工具
- [ ] 在 Cursor 中設定 MCP 連線
- [ ] 測試：讓 AI 透過 MCP 查詢你的 `auth` 模組

### Week 3：自動化與進階
- [ ] 加入 Husky Git Hooks
- [ ] 實作增量同步邏輯
- [ ] 加入其他語言適配器（Python、Go、C）
- [ ] 建立 `module_overrides` 覆寫機制
- [ ] 撰寫文件，讓團隊成員能上手

### 關鍵成功因素
1. **從一個模組開始**：先只掃描 `auth` 模組，跑通整個流程，再擴展到其他模組。
2. **MCP 工具要封裝**：千萬不要暴露 `execute_sql`，一定要用高階語意工具。
3. **指紋機制不能省**：這是避免幽靈資料的關鍵。
4. **雙 DB 權限要嚴格**：Global DB 必須在 MCP 層級設定為唯讀。

## 🛑 關鍵技術謬誤（需修正的 Bug）

### 1. 跨庫附加的嚴重錯誤（Phase 2, Step 2.1）

* **指南寫法**：
```typescript
const globalDb = new Database('.codegraph/global_modules.db', { readonly: true });
const localDb = new Database('.codegraph/project_local.db');
localDb.exec(`ATTACH '.codegraph/global_modules.db' AS global;`);

```


* **謬誤所在**：
當你把 `global_modules.db` 先用 `readonly: true` 獨立開啟成一個 `globalDb` 連線，**接著**又試圖在 `localDb` 連線中使用 `ATTACH` 語句去掛載同一個檔案。在 SQLite 的 WAL 或鎖定機制下，同一個進程對同一個檔案重複開啟不同的連線，非常容易觸發 `SQLITE_BUSY` 鎖定錯誤。
更嚴重的問題是：**你在下方 `handleSearch` 工具中，依然在使用 `globalDb.prepare(...)` 進行查詢**。既然已經 `ATTACH AS global`，就應該統一使用 `localDb` 進行跨庫查詢，否則 `ATTACH` 這行完全是多餘的。
* **修正方案**：
精簡連線，只保留一個 `localDb` 主連線，並透過它處理所有跨庫事務：
```typescript
const localDb = new Database('.codegraph/project_local.db');
// 直接由 localDb 附加 global，不需要獨立宣告 globalDb 變數
localDb.exec(`ATTACH DATABASE '.codegraph/global_modules.db' AS global;`);

```



### 2. Full-Text Search (FTS5) 外部內容表的觸發器遺漏（Phase 1, Step 1.1）

* **指南寫法**：
使用 `content='backend_apis'` 建立了外部內容表（External Content Table）。
* **謬誤所在**：
在 SQLite 中，`content='tableName'` 的 FTS5 虛擬表**只是一個影子**。當你向 `backend_apis` 插入（INSERT）或更新資料時，**FTS 索引並不會自動更新**！如果在 `scanDirectory` 裡只寫入 `backend_apis`，你的 FTS 表會一直是空的，`MATCH` 查詢會查不到任何東西。
* **修正方案**：
必須在初始化資料庫時，加上 SQL `TRIGGER`（觸發器），讓資料同步更新至 FTS 虛擬表：
```sql
-- 當 backend_apis 有新資料時，同步寫入 FTS
CREATE TRIGGER IF NOT EXISTS ba_ai AFTER INSERT ON backend_apis BEGIN
  INSERT INTO backend_apis_fts(rowid, module_name, api_name, signature, description) 
  VALUES (new.id, new.module_name, new.api_name, new.signature, new.description);
END;

-- 當 backend_apis 被刪除時，同步移出 FTS
CREATE TRIGGER IF NOT EXISTS ba_ad AFTER DELETE ON backend_apis BEGIN
  INSERT INTO backend_apis_fts(backend_apis_fts, rowid, module_name, api_name, signature, description) 
  VALUES('delete', old.id, old.module_name, old.api_name, old.signature, old.description);
END;

```



### 3. FTS5 語法與跨庫附加後的查詢衝突（Phase 2, Step 2.1）

* **指南寫法**：
```typescript
const globalResults = globalDb.prepare(`SELECT ... FROM backend_apis_fts WHERE backend_apis_fts MATCH ?`).all(keyword);

```


* **謬誤所在**：
1. 如前所述，若改用 `ATTACH` 方案，全域查詢語法必須指定資料庫前綴：`global.backend_apis_fts`。
2. 核心錯誤：`WHERE backend_apis_fts MATCH ?` 在 SQLite FTS5 的標準標準中，如果隱式地將表名作為欄位名傳給 `MATCH`，有時會引發語法歧義。標準且防呆的寫法應該是「虛擬表名 MATCH 關鍵字」，或者使用 `backend_apis_fts MATCH 'field:' || ?`。


* **修正方案**（統一由 `localDb` 跨庫查詢）：
```typescript
// 查詢本地
const localResults = localDb.prepare(`
  SELECT id, module_name, api_name, signature, description
  FROM backend_apis_fts 
  WHERE backend_apis_fts MATCH ?
`).all(keyword);

// 查詢跨庫附加的全域公共模組 (注意 global. 前綴)
const globalResults = localDb.prepare(`
  SELECT id, module_name, api_name, signature, description
  FROM global.backend_apis_fts 
  WHERE global.backend_apis_fts MATCH ?
`).all(keyword);

```



### 4. 增量同步模式下的全量刪除邏輯 Bug（Phase 1, Step 1.3）

* **指南寫法**：
```typescript
if (mode === 'full') { ... }
// 下方無論什麼 mode，都直接執行：
await scanDirectory('global-modules', globalDb, tsAdapter);
await scanDirectory('src', localDb, tsAdapter);

```


* **謬誤所在**：
你的 `build-graph.ts` 雖然傳入了 `mode = 'incremental'`（增量），但是在 `scanDirectory` 函數中，你依然呼叫了 `getAllFiles(dir)` 掃描了**整個目錄的所有檔案**。這意味著你的「增量同步」在代碼實作中**其實還是全量掃描**。
如果在大專案中，每次 Commit 都要花數十秒去完整掃描數千個檔案，Husky Hook 就會被開發者強行拔掉（`--no-verify`）。
* **修正方案**：
修改 `main` 函數，如果模式是 `incremental`，必須透過 `git diff` 或是檔案修改時間（mtime）只傳入變更的檔案路徑給 `scanDirectory`。

### 5. 指紋（Fingerprint）計算的致命缺陷（Phase 1, Step 1.3）

* **指南寫法**：
```typescript
const fingerprint = crypto.createHash('md5')
  .update(`${file}:${api.line_start}:${api.api_name}:${api.signature}`)
  .digest('hex');

```


* **謬誤所在**：
你在指紋中加入了 `api.line_start`（開始行號）。這是一個隱形炸彈！
假設你在檔案的**第一行**加了一個空行（Enter），該檔案內隨後所有的 API **行號全部都會 +1**。這會導致這家檔案裡所有 API 的指紋「全部改變」，進而在執行 `INSERT OR REPLACE` 時，因為主鍵（`module_name`, `api_name`）觸發取代，但舊的指紋和行號歷史可能在關聯表（如 `feature_contracts`）中產生幽靈錯位。
指紋的目的是偵測「代碼內容（實質邏輯）是否有變更」，絕對不能把「行號」或「檔案路徑」算進內容指紋中。
* **修正方案**：
指紋應該只針對 **「AST 節點的文字內容（raw text）」** 進行雜湊：
```typescript
// TypeScriptAdapter 萃取時應將 node.text 一併傳出
const fingerprint = crypto.createHash('md5')
  .update(api.api_body_text) // 整個 Function 的原始代碼字串
  .digest('hex');

```



### 6. Node.js ES Modules (ESM) 與 `ts-node` 的環境衝突（Phase 1, Step 1.4）

* **指南寫法**：
在 `scripts/mcp-server/index.ts` 中使用了 `import { Server } from '@modelcontextprotocol/sdk/server/index.js';`（這是標準的 ESM 寫法）。但在 `package.json` 卻直接用 `ts-node` 執行。
* **謬誤所在**：
`ts-node` 預設執行在 CommonJS 環境下，當它遇到 `@modelcontextprotocol/sdk` 的 `.js` 後綴引用的 ESM 模組時，會直接拋出 `ERR_REQUIRE_ESPNOT_SUPPORTED` 或 `Cannot use import statement outside a module` 錯誤，導致 MCP Server 完全無法啟動。
* **修正方案**：
建議使用現今效能更好、原生支援 ESM 轉換的 `tsx` 取代 `ts-node`：
```bash
npm install -D tsx

```


`package.json` 改為：
```json
"scripts": {
  "build-graph": "tsx scripts/build-graph.ts full",
  "sync-graph": "tsx scripts/build-graph.ts incremental",
  "mcp-server": "tsx scripts/mcp-server/index.ts"
}

```

## 💡 遺漏的關鍵議題（需要補充的架構思維）

### 1. 遺漏：增量同步時的「孤兒節點（Orphan Nodes）」清理機制

* **痛點**：當開發者在 `src/auth.ts` 中**刪除**了一個名為 `oldLogout` 的函數並儲存。
* **目前的盲點**：你的代碼是用 Tree-sitter 去掃描「現存的程式碼」。既然函數已經被刪了，Tree-sitter 就不會解析到它，它就不會出現在 `apis` 陣列中。那麼進了 `db.prepare(... INSERT OR REPLACE)` 時，**因為這筆資料根本沒被丟進去，SQLite 完全不會動到它，它會變成幽靈資料（孤兒節點）永久留在 DB 中**，導致 AI 繼續瞎猜。
* **補充實作**：
在增量同步某個檔案前，必須先根據 `file_path` 將該檔案在 DB 中舊有的 API 記錄**全部清空**，再把 Tree-sitter 掃描到的新節點寫入。
```typescript
// 在掃描單一檔案前執行：
db.prepare(`DELETE FROM backend_apis WHERE file_path = ?`).run(file);

```



### 2. 遺漏：Tree-sitter 記憶體與異步載入的效能陷阱

* **痛點**：`web-tree-sitter` 的 WASM 模組在 Node.js 中執行時，其核心是 C/C++ 的底層記憶體。如果在迴圈中頻繁解析大量檔案（如 `getAllFiles` 掃出幾百個檔案），卻沒有手動釋放樹節點，會導致 V8 記憶體迅速暴漲。
* **補充實作**：
在解析完一個檔案後，必須手動調用 `tree.delete()` 來釋放 WASM 的記憶體堆疊：
```typescript
extractAPIs(filePath: string, sourceCode: string): ExtractedAPI[] {
  const tree = this.parser.parse(sourceCode);
  const apis: ExtractedAPI[] = [];
  // ... 走訪與萃取邏輯 ...

  tree.delete(); // 🌟 關鍵：一定要 delete 釋放記憶體，否則全量掃描時專案大一點會 OOM 崩潰
  return apis;
}

```



### 3. 遺漏：關聯表（Contracts / Dependencies）的自動資料來源

* **痛點**：在你的 Schema 中設計了非常棒的 `module_dependencies`、`feature_contracts`（前後端合約）與 `communication_contracts`（MQTT 協定）。然而，在 Phase 1 的掃描腳本中，**你只寫了抓取 `backend_apis` 的邏輯**，其他表全是空的。
* **補充思維**：
如果這些合約完全靠開發者「手動填入 SQLite」，基本上兩天後就不會有人維護了。
* **依賴關係（Dependencies）**：應該在 TypeScript 快適配器中，解析 `import_statement` 的路徑，自動寫入 `module_dependencies` 表。
* **MQTT 合約（Communication）**：在 IoT 專案中，通常會有 `mqtt.publish("telemetry/temperature", payload)` 這樣的代碼。你應該在適配器中寫一條 Tree-sitter Query，專門去抓 `call_expression` 中方法名為 `publish` 或 `subscribe` 的節點，將其中的第一個參數（Topic）自動萃取並沉澱到 `communication_contracts` 表中。



### 4. 遺漏：唯讀 Global DB 的發佈與更新機制

* **痛點**：架構圖上規劃 `global_modules.db` 是進 Git 的公共資產。但如果是多專案團隊，每個人都在本地修改自己的核心庫，Git 衝突（Binary conflict）會非常痛苦（因為 SQLite 是二進位檔案，無法進行 Git Merge）。
* **補充建議**：
不要將 `global_modules.db` 直接放進專案 Git 倉庫。應該將你的公共公共模組（`global-modules/`）獨立成一個內部 Git 倉庫，並利用 CI/CD 腳本在遠端打包生成 `global_modules.db`，最後將這個 `.db` 檔作為一個 **npm 私有套件（或是 Release 資產）** 讓各個業務專案下載。
專案的 `package.json` 只需要鎖定這個知識庫的版本即可（例如 `"@your-company/iot-knowledge-graph": "1.2.0"`）。

### 🛠️ 修正二：`TypeScriptAdapter` (記憶體釋放、指紋修正、自動萃取關聯表)

針對您指出的 **Bug #5 (指紋炸彈)**、**遺漏 #2 (WASM OOM)** 與 **遺漏 #3 (關聯表自動來源)**。

```typescript
// scripts/adapters/typescript-adapter.ts
import Parser from 'web-tree-sitter';

export interface ExtractedAPI {
  universal_kind: string;
  api_name: string;
  signature: string;
  description: string;
  file_path: string;
  line_start: number;
  line_end: number;
  raw_text: string; // 🌟 修正 Bug #5：用於計算純內容指紋
}

export interface ExtractedDependency {
  source_module: string;
  target_module: string;
  import_path: string;
}

export interface ExtractedMqttContract {
  protocol: string;
  topic: string;
  payload_example: string;
}

export class TypeScriptAdapter {
  private parser: Parser;
  
  async init() {
    await Parser.init();
    this.parser = new Parser();
    const Lang = await Parser.Language.load('node_modules/tree-sitter-wasms/out/tree-sitter-typescript.wasm');
    this.parser.setLanguage(Lang);
  }
  
  extractAPIs(filePath: string, sourceCode: string, moduleName: string): { apis: ExtractedAPI[], deps: ExtractedDependency[], mqtt: ExtractedMqttContract[] } {
    const tree = this.parser.parse(sourceCode);
    const apis: ExtractedAPI[] = [];
    const deps: ExtractedDependency[] = [];
    const mqtt: ExtractedMqttContract[] = [];
    
    const cursor = tree.walk();
    this.traverse(cursor, (node) => {
      // 1. 萃取 API (函數/類別)
      if (node.type === 'function_declaration' || node.type === 'class_declaration') {
        const nameNode = node.childForFieldName('name');
        apis.push({
          universal_kind: node.type === 'function_declaration' ? 'Function' : 'Class',
          api_name: nameNode?.text || 'anonymous',
          signature: this.getSignature(node),
          description: this.extractJSDoc(node),
          file_path: filePath,
          line_start: node.startPosition.row,
          line_end: node.endPosition.row,
          raw_text: node.text // 🌟 修正 Bug #5：保留原始碼文字用於指紋
        });
      }
      
      // 2. 🌟 遺漏 #3：自動萃取 Dependencies (import 語句)
      if (node.type === 'import_statement') {
        const pathNode = node.childForFieldName('source');
        if (pathNode) {
          const importPath = pathNode.text.replace(/['"]/g, '');
          // 簡單啟發式規則：將相對路徑轉換為模組名
          const targetModule = importPath.startsWith('.') ? importPath.split('/').slice(-2)[0] : importPath.split('/')[0];
          deps.push({
            source_module: moduleName,
            target_module: targetModule,
            import_path: importPath
          });
        }
      }

      // 3. 🌟 遺漏 #3：自動萃取 MQTT Contracts (publish/subscribe)
      if (node.type === 'call_expression') {
        const funcNode = node.childForFieldName('function');
        if (funcNode?.text.includes('publish') || funcNode?.text.includes('subscribe')) {
          const args = node.childForFieldName('arguments');
          if (args && args.childCount > 0) {
            const topicNode = args.children.find(c => c.type === 'string');
            if (topicNode) {
              mqtt.push({
                protocol: 'MQTT',
                topic: topicNode.text.replace(/['"]/g, ''),
                payload_example: args.children[1]?.text || '{}'
              });
            }
          }
        }
      }
    });

    // 🌟 修正 遺漏 #2：手動釋放 WASM 記憶體，防止 OOM
    tree.delete(); 
    
    return { apis, deps, mqtt };
  }

  // ... (省略 getSignature, extractJSDoc, traverse 等輔助方法)
}
```

```typescript
    // 🌟 修正 Bug #4：透過 git diff 只取得變更的檔案
    try {
      const diffOutput = execSync('git diff --name-only HEAD', { encoding: 'utf-8' });
      filesToProcess = diffOutput.split('\n').filter(f => f.startsWith('src/') && f.endsWith('.ts'));
    } catch (e) {
      console.warn('⚠️ 無法執行 git diff，降级為全量掃描');
      filesToProcess = getAllFiles('src');
    }
  }

  for (const file of filesToProcess) {
    // 🌟 修正 遺漏 #1：掃描前先清空該檔案的舊記錄 (清理孤兒節點)
    localDb.prepare('DELETE FROM backend_apis WHERE file_path = ?').run(file);
    localDb.prepare('DELETE FROM module_dependencies WHERE source_module = ?').run(extractModuleName(file));

    if (!fs.existsSync(file)) continue; // 檔案被刪除的情況

    const content = fs.readFileSync(file, 'utf-8');
    const moduleName = extractModuleName(file);
    const { apis, deps, mqtt } = adapter.extractAPIs(file, content, moduleName);

    // 寫入 APIs
    for (const api of apis) {
      // 🌟 修正 Bug #5：指紋只計算 raw_text (實質內容)，不包含行號
      const fingerprint = crypto.createHash('md5').update(api.raw_text).digest('hex');
      
      localDb.prepare(`
        INSERT INTO backend_apis 
        (module_name, language, universal_kind, api_name, signature, description, file_path, line_start, line_end, fingerprint)
        VALUES (?, 'ts', ?, ?, ?, ?, ?, ?, ?, ?)
      `).run(moduleName, api.universal_kind, api.api_name, api.signature, api.description, api.file_path, api.line_start, api.line_end, fingerprint);
    }

    // 寫入 Dependencies
    for (const dep of deps) {
      localDb.prepare(`INSERT OR IGNORE INTO module_dependencies (source_module, target_module, import_path) VALUES (?, ?, ?)`)
        .run(dep.source_module, dep.target_module, dep.import_path);
    }

    // 寫入 MQTT Contracts
    for (const m of mqtt) {
      localDb.prepare(`INSERT OR IGNORE INTO communication_contracts (protocol, topic_or_path, payload_schema, provider_lang) VALUES (?, ?, ?, 'ts')`)
        .run(m.protocol, m.topic, m.payload_example);
    }
  }
  
  console.log(`✅ 同步完成！共處理 ${filesToProcess.length} 個檔案。`);
}

// ... (省略 getAllFiles, extractModuleName)
main().catch(console.error);
```

### 💡 針對「遺漏 #4：Global DB 發佈機制」的架構建議

您提到的 **Binary Git Conflict** 問題非常精準。將 `.db` 檔直接放進業務專案的 Git 是反模式。

**最終建議的企業級落地架構：**
1. **獨立知識庫 Repo**：建立一個 `@your-company/iot-knowledge-base` 的獨立 Git Repo，裡面只有 `global-modules/` 原始碼與 `build-graph.ts`。
2. **CI/CD 自動打包**：在該 Repo 設定 GitHub Actions，當 `main` 分支有 push 時，自動執行 `npm run build-graph`，並將產出的 `global_modules.db` 發布為 **GitHub Release 資產**，或是發布到內部的 **npm Registry** (例如 `@your-company/iot-graph-db@1.0.4`)。
3. **業務專案引用**：業務專案的 `package.json` 只需依賴這個套件：
   ```json
   "dependencies": {
     "@your-company/iot-graph-db": "^1.0.4"
   }
   ```
4. **初始化腳本**：業務專案的 `postinstall` 腳本自動從 `node_modules/@your-company/iot-graph-db/dist/global_modules.db` 複製一份到 `.codegraph/global_modules.db`。

這樣就完美解決了版本控制衝突，且能確保所有業務專案使用的公共知識庫版本是鎖定且一致的！
