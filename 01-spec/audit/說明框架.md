要將 Spring Boot 龐大的生態系從「零碎片段」整合成「完整的知識圖譜」，最關鍵的心法是：**不要從「API 記憶」的角度去學，而是要從「靜態架構（組裝）」與「動態運行（生命週期）」這兩個維度去建立心智模型。**

以下我為您梳理一份 **Spring Boot 核心知識圖譜**，幫助您把之前學過的 IoC、AOP，以及您上傳的審計日誌程式碼，全部安放到正確的位置上。

### 第一維度：靜態架構圖譜（Spring 的三大引擎）

這是 Spring 的「底層物理定律」，所有上層的業務功能都是建立在這些引擎之上。

#### 1. 核心引擎：IoC 容器 (Inversion of Control)
*   **概念**：物件的創建與依賴關係管理。
*   **圖譜位置**：最底層。所有的 `@Component`, `@Service`, `@Repository`, `@Controller` 都是被這個引擎掃描並實例化為 Bean。
*   **您的程式碼映射**：`AuditAsyncConfig` 中的 `@Bean("auditExecutor")` 就是手動向這個引擎註冊一個執行緒池物件。

#### 2. 擴展引擎：AOP (Aspect-Oriented Programming)
*   **概念**：在不修改原始碼的情況下，動態攔截並增強方法。
*   **圖譜位置**：環繞在 IoC 容器之上，作為「外掛系統」存在。
*   **您的程式碼映射**：`BaseLoggerAspect` 就是掛載在這個引擎上的外掛，它攔截了帶有 `@AuditEvent` 的方法。

#### 3. 啟動引擎：Spring Boot Auto-Configuration (自動配置)
*   **概念**：這是 Spring Boot 有別於傳統 Spring 的靈魂。它透過 `@Conditional` 系列註解，根據 classpath 下的 jar 包，**自動**幫您配置好 IoC 和 AOP 所需的 Bean。
*   **圖譜位置**：應用啟動的最外層。
*   **白話理解**：傳統 Spring 需要您手動寫幾百行 XML 或 Java Config 來配置資料庫連線池；Spring Boot 只要您引入 `spring-boot-starter-data-jpa`，啟動引擎就會自動幫您把 JPA 相關的 Bean 全部組裝好。

### 第二維度：動態生命週期圖譜（串聯碎片的最強武器）

很多時候我們覺得知識是碎片的，是因為我們只看了「靜態的類別」，沒看到「動態的請求流向」。
**把一個 HTTP 請求從進入到返回的完整生命週期畫出來，您就能把 80% 的 Spring 組件串聯起來。**

以下是標準的 Spring Boot Web 請求生命週期圖譜：

```text
[客戶端 Request] 
   │
   ▼
1. 【Filter Chain (過濾器鏈)】 (Servlet 容器層，如 Tomcat)
   ├─ 處理最外層邏輯：CORS 跨域、字元編碼、Spring Security 認證 (JWT 解析)。
   │
   ▼
2. 【DispatcherServlet】 (Spring MVC 的核心前端控制器)
   ├─ 所有請求的總入口，負責將請求分發給對應的 Controller。
   │
   ▼
3. 【HandlerInterceptor (攔截器)】 (Spring MVC 層)
   ├─ 處理 Controller 執行前後的邏輯：權限校驗、多租戶 TenantContext 初始化、請求日誌。
   │
   ▼
4. 【Controller (控制器)】 (展現層)
   ├─ 接收參數、參數校驗 (@Valid)、呼叫 Service、返回 DTO。
   │
   ▼
5. 【AOP Aspect (切面)】 (Spring AOP 層) ⭐️ 您的程式碼在這裡
   ├─ 攔截 Service 方法。例如：@Transactional (事務管理)、@AuditEvent (審計日誌)。
   │
   ▼
6. 【Service (業務邏輯層)】 (核心業務層)
   ├─ 處理核心業務邏輯。例如：您的 AuditAsyncWriter 觸發非同步寫入。
   │
   ▼
7. 【Repository (資料存取層)】 (持久層)
   ├─ Spring Data JPA / MyBatis。執行 SQL，與資料庫互動。
   │
   ▼
[資料庫 DB] -> 原路返回 Response 給客戶端
```

**💡 圖譜解析：**
當您理解了這個流程，您就不會再把 Filter、Interceptor、AOP 搞混了：
*   **Filter** 在 Spring 之外（Tomcat 層），拿不到 Spring 的 Bean，適合做粗粒度的安全認證。
*   **Interceptor** 在 Spring MVC 層，能拿到 Handler (Controller) 的資訊，適合做細粒度的權限和上下文（如您的 `TenantContext`）。
*   **AOP** 在 Spring 核心層，能攔截到**任何 Bean 的方法**（包含 Service），適合做事務、審計、快取等與具體業務邏輯深度結合的橫切關注點（如您的 `BaseLoggerAspect`）。

### 第三維度：將您的「審計日誌」放入知識圖譜

現在，我們把您上傳的三個檔案，放進上面的動態生命週期圖譜中，看看它們是如何協作的：

1.  **請求進入**：前端呼叫了一個帶有 `@AuditEvent` 註解的 API。
2.  **觸發 AOP (`BaseLoggerAspect`)**：
    *   在 Controller/Service 方法**執行前**，AOP 在主執行緒抓取 `TenantContext`、`SecurityContext`、IP 等上下文（因為非同步執行緒拿不到這些 ThreadLocal 變數）。
    *   執行目標方法（業務邏輯）。
    *   在方法**執行後 (finally)**，AOP 計算耗時，並將收集到的參數打包。
3.  **觸發非同步 (`AuditAsyncWriter`)**：
    *   AOP 呼叫 `auditAsyncWriter.saveAsync()`。
    *   因為有 `@Async("auditExecutor")`，Spring 將這個任務丟給 `AuditAsyncConfig` 配置的專屬執行緒池。
    *   因為有 `@RunInSystemTenantContext`，新執行緒啟動後，AOP 會自動將該執行緒的上下文切換為 SYSTEM，確保能跨租戶查詢使用者資訊並寫入日誌。
4.  **完成寫入 (`UserEventLogRepository`)**：最終透過 JPA 將日誌存入資料庫。

**結論**：這三個檔案完美展示了 Spring 中 **AOP 攔截 -> IoC 依賴注入 -> 非同步執行緒池 -> 多租戶上下文切換** 的完整協作鏈路。

### 如何自主構建與鞏固您的「知識圖譜」？

要擺脫零碎片段，建議您在後續學習或維護專案時，採用以下三個方法：

1.  **畫「請求流向圖」而非「類別圖」**：
    不要只畫 A 類別依賴 B 類別。試著畫出：「當使用者點擊『建立訂單』按鈕時，HTTP 請求經過了哪些 Filter？哪個 Interceptor？哪個 AOP？最後進了哪個 Queue？」**動態的流向能瞬間打通任督二脈。**
2.  **善用 Debug 追蹤「魔法」的邊界**：
    Spring 最神奇的地方在於「它幫您做了什麼」。在 IDE 中打上中斷點，特別是在 `DispatcherServlet.doDispatch()` 或是您的 AOP 切面中，單步執行 (Step Into)，親眼看看 Spring 是如何把一個簡單的 `@Async` 轉換成代理物件 (Proxy) 並丟入執行緒池的。
3.  **理解「約定優於配置 (Convention over Configuration)」**：
    Spring Boot 的很多碎片知識（如各種 Starter、預設端口、預設資料庫連線池）都是基於「約定」。當您遇到不理解的配置時，問自己：「Spring Boot 預設幫我做了什麼？我現在為什麼要覆蓋它的預設值？」

**總結來說：**
IoC 是**骨架**（負責組裝），AOP 是**神經系統**（負責攔截與增強），Spring Boot AutoConfig 是**皮膚與衣服**（負責快速包裝與啟動），而 HTTP 請求生命週期則是**血液循環**（讓整個系統活起來）。把這四個概念記住，您就擁有了一張完整的 Spring Boot 知識圖譜。
