## 🛠️ 建議補充（可讓模板更完善）

### 1. ⚠️ AC 編號格式不明確（可能導致與 FC 編號混淆）

**問題陳述**：  
模板中 AC 編號使用 `AC-1`、`AC-2` 的格式，但**沒有定義 AC 編號與 FC（功能群）編號的關係**。若使用者有 6 個 FC，每個 FC 內有 3~5 個 AC，則 AC 編號可能重複（例如 F1 有 AC-1，F2 也有 AC-1），導致追溯性混亂。

**影響**：  
AI 在生成測試時，可能產生 `AC_1_markAsRead_xxx` 與 `AC_1_createAnnouncement_xxx` 這樣的重複編號，破壞追溯性。

**修正建議**：

將 AC 編號格式改為 **`[FC編號]_[流水號]`**，例如：

```markdown
### 3.1 快樂路徑 (Happy Path)
- [ ] **F1-AC1**：使用者所屬部門有 3 筆可見公告...
- [ ] **F1-AC2**：使用者已讀其中 1 筆，`isRead=true`...
```

**對應測試命名**：`F1_AC1_listVisible_returns3Announcements`

**模板修改**：

```markdown
## 3. 功能完成條件（測試合約 / Acceptance Criteria）

> AC 編號格式：`[FC編號]-AC[流水號]`，例如 `F1-AC1`、`F3-AC7`。
> 流水號從 1 開始，每個 FC 獨立編號。

### 3.1 快樂路徑
- [ ] **F1-AC1**：...
- [ ] **F1-AC2**：...
```

---

### 2. ⚠️ AC 描述缺乏「測試層級歸屬」標示

**問題陳述**：  
目前模板沒有明確要求使用者在撰寫 AC 時，標示「這個 AC 該由哪一層的測試負責」（Service 單元測試 / Controller 整合測試 / Repository 整合測試）。這可能導致 AI 在 Service 單元測試中試圖驗證 Controller 層的行為（如未登入攔截）。

**影響**：  
AI 可能產生在錯誤測試層級驗證的 AC，導致測試設計混亂。

**修正建議**：

在 AC 清單中加入 **「測試層級」** 欄位，或在 AC 標題後以括號標示：

```markdown
### 3.3 安全性與副作用
- [ ] **F3-AC7（Service 單元測試 + 整合測試）**：租戶隔離 — 必須使用 `TenantContext.getCurrentTenantId()` 的值傳遞給 Repository，且單元測試須以 `ArgumentCaptor` 驗證此參數。
- [ ] **F1-AC11（Controller 整合測試）**：`GET /v1/auth/announcements` 需 `isAuthenticated()`；未登入請求應被 Spring Security 攔截。
- [ ] **F2-AC14（Controller 層整合測試）**：`expireAt` 早於 `publishAt` 時，DTO 的 `@AssertTrue` 應觸發驗證失敗。
```

**可接受的縮寫**：
- `（Service 單元測試）` — 預設，可省略
- `（Controller 整合測試）`
- `（Repository 整合測試）`
- `（Service 單元測試 + 整合測試）`

---

### 3. 📝 建議補充「AC 描述結構化範例」

**問題陳述**：  
模板中 AC 的範例雖然具體，但沒有強制要求使用 **GIVEN-WHEN-THEN** 結構。對於較複雜的 AC，AI 可能因描述不夠結構化而難以產生精確的測試。

**影響**：  
若使用者寫出「系統應正常運作」這類模糊描述，AI 仍可能產生無效測試。

**修正建議**：

在章節 3 的開頭補充 AC 撰寫指南：

```markdown
## 3. 功能完成條件（測試合約 / Acceptance Criteria）

> **AC 撰寫指南**：
> - 每一條 AC 必須是**可驗證的具體情境**，避免模糊用語（如「系統要穩」）。
> - 建議採用 **GIVEN-WHEN-THEN** 結構撰寫：
>   - **GIVEN**：前置條件（例如「使用者有 3 筆未讀公告」）
>   - **WHEN**：執行動作（例如「呼叫 markAllAsRead」）
>   - **THEN**：預期結果（例如「成功寫入 3 筆閱讀記錄」）
> 
> **範例**：
> - ❌ 模糊：`系統應能正確標記已讀。`
> - ✅ 具體：`GIVEN 使用者有 3 筆未讀公告，WHEN 呼叫 markAllAsRead，THEN 成功寫入 3 筆閱讀記錄，且不拋出例外。`
```

---

### 4. 📝 建議補充「例外行為」欄位（相依元件表格）

**問題陳述**：  
目前「相依元件」表格只要求寫「預期行為（正常回傳值）」，但沒有要求寫「當外部元件拋出例外時，Service 該如何反應」。這可能導致 AI 遺漏 `try-catch` 或例外轉譯邏輯。

**影響**：  
若 AI 僅根據正常行為實作，遇到 DB 連線失敗等例外時，Service 可能未妥善處理。

**修正建議**：

在表格中加入「例外行為」欄位（可選填）：

```markdown
## 4. 相依元件與對外介面

| 元件 | 方法 | 預期行為（正常） | 例外行為（Mock 拋出時） |
| :--- | :--- | :--- | :--- |
| `AnnouncementRepository` | `countUnread(userId, deptId, now)` | 回傳未讀公告總數（Long） | 若拋出 `DataAccessException`，Service 不攔截，由全局例外處理器處理 |
| `AnnouncementReadRepository` | `markAllAsRead(userId, tenantId, deptId)` | void（執行批次插入） | 若拋出 `DataAccessException`，Service 不攔截 |
```

---

### 5. 📝 建議補充「單元測試 vs 整合測試」的判斷指引

**問題陳述**：  
模板雖然有「整合測試規劃」章節，但沒有明確說明「什麼條件下該用整合測試」，使用者可能難以判斷哪些 AC 需要標示為整合測試。

**影響**：  
使用者可能將所有 AC 都寫成單元測試，導致關鍵的 SQL / CASCADE / Filter 行為未被驗證。

**修正建議**：

在「整合測試規劃」章節開頭補充判斷指引：

```markdown
## 6. 整合測試規劃（Integration Test Strategy）

> **何時該用整合測試？**
> 
> 當 AC 涉及以下任一情況時，**必須**使用整合測試（`@SpringBootTest` 或 `@DataJpaTest`），因為單元測試（Mockito）無法模擬：
> 
> 1. **資料庫行為**：SQL 語法、Hibernate `@Filter`、`ON DELETE CASCADE`、`@Version` 樂觀鎖。
> 2. **Native SQL**：`ON CONFLICT`、CTE、窗口函數等 PostgreSQL 專屬語法。
> 3. **檔案 IO**：檔案儲存、刪除、讀取（需 `@TempDir`）。
> 4. **Transaction 行為**：`@Transactional` 滾回、Propagation 等。
> 5. **Spring Security 攔截**：未登入、權限不足等 Controller 層行為。
```
---

### 6. 📝 建議在開頭補充「FC 編號」定義欄位

**問題陳述**：  
模板中使用了 `[功能模組名稱]` 作為標題，但沒有定義「FC 編號」（功能群編號）的填寫位置。若使用者要為多個 FC 各填一份文件，無法區分不同 FC。

**影響**：  
AI 可能將多份文件的 AC 編號混淆。

**修正建議**：

在標題下方新增「FC 編號」欄位：

```markdown
# [功能模組名稱] 需求規格書

> **FC 編號**：______（例如：F1, F2, F3...）
> **所屬專案**：______
```
