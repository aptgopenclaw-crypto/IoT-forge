### 📝 實務上，你可以在需求文件中這樣「合併」撰寫（範例）

> **需求描述（含 Use Case 與完成條件）：**
> 
> 實作「全部標為已讀」功能。當使用者在公告列表點擊「全部已讀」時，系統應批次將目前租戶內、且該使用者有權限看到的所有未讀公告，寫入閱讀紀錄表。
> 
> **必須滿足的測試驗收條件：**
> 1. **邏輯正確**：僅作用於 `status='PUBLISHED'` 且未過期的公告。
> 2. **租戶嚴格隔離**：即使 SecurityContext 中有其他租戶資訊，Service 層**必須**使用 `TenantContext.getCurrentTenantId()` 傳給 Repository（寫單元測試時請用 `ArgumentCaptor` 驗證此參數）。
> 3. **防禦性設計**：若目前沒有任何未讀公告，系統應優雅結束（回傳成功或 0 筆異動），不得拋出 `NullPointerException` 或 `EmptyResultDataAccessException`。
> 4. **冪等性**：使用者短時間內連點 3 次，不得產生重複閱讀紀錄（依賴資料庫 UNIQUE 約束）。

- **對你（PM / 開發者）**：在 Code Review 時，你不需一行一行看程式碼，只要確認 `測試程式碼` 是否涵蓋了你文件中的第 1~4 點「完成條件」。如果涵蓋了，這支程式的品質防線就建立了。

### ⚠️ 一個小小的語言微調（讓 AI 更聽話）

建議將 **「功能完成條件」** 在文件中改名為 **「測試合約 (Testing Contract)」** 或 **「驗收門檻 (Acceptance Thresholds)」**。

因為「完成條件」聽起來像「做完就好」，而「測試合約」會強烈暗示 AI：**「這裡的每一條，都必須轉換成一個或多個 `@Test` 方法來驗證，否則視為未完成。」**

你的想法完全走在正確的軟體工程道路上。**需求敘述給 Context，Use Case 給流程，功能完成條件（測試合約）給邊界與安全規則**。只要這三者在文件中明確，你完全不需要去思考「要 mock 哪些 method」，AI 會根據這份合約自動幫你補齊所有的單元測試。
