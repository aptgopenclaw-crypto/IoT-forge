這是一個非常敏銳的觀察！這三個 class 雖然看起來職責各不相同（一個管 Session、一個管密碼政策、一個管 JWT 解析），但它們被集中放在 `com.taipei.iot.auth.port` 這個 package 中，是因為它們在系統架構中扮演著完全相同的角色：**「適配器 (Adapter) / 橋接器 (Bridge)」**。

這背後隱藏著一個非常經典且優秀的架構設計模式：**端口與適配器架構 (Ports and Adapters / 六邊形架構)** 以及 **依賴反轉原則 (DIP)**。

以下為您詳細剖析為什麼要這樣設計，以及這個 package 存在的意義：

---

### 1. 這三個 Class 的共同本質：介面的實作 (Adapter)

如果您仔細看它們的程式碼，會發現一個共同規律：
* **`AuthSessionRevoker`**：實作了 `com.taipei.iot.common.auth.port.SessionRevoker` 介面，並委派給內部的 `UserSessionService`。
* **`AuthPasswordPolicyProvider`**：實作了 `com.taipei.iot.common.auth.port.PasswordPolicyProvider` 介面，並委派給內部的 `PasswordPolicyResolver`。
* **`AuthTokenJtiReader`**：實作了 `com.taipei.iot.common.auth.port.TokenJtiReader` 介面，並委派給內部的 `JwtUtil`。

**結論**：它們都不是 `auth` 模組內部的核心業務邏輯，而是**專門為了「滿足其他模組的依賴」而存在的橋接層**。

---

### 2. 為什麼要這樣設計？（核心架構原因）

在大型多模組專案中，我們經常遇到一個問題：**模組間的依賴關係該如何管理？**

假設系統中有一個 `asset` (資產管理) 模組，它在進行某些敏感操作時，需要「撤銷使用者的 Session」或「檢查密碼政策」。

#### ❌ 錯誤的做法（高耦合 / 循環依賴）
如果 `asset` 模組直接依賴 `auth` 模組的 `UserSessionService`：
1. `asset` 會被迫引入整個 `auth` 模組的依賴。
2. 如果未來 `auth` 模組重構，改了 `UserSessionService` 的方法簽名，`asset` 模組也會跟著崩潰。
3. 更糟的是，如果 `auth` 模組也需要用到 `asset` 的某些功能，就會產生**循環依賴 (Circular Dependency)**，導致專案無法編譯。

#### ✅ 正確的做法（依賴反轉 / 端口與適配器）
為了解耦，系統將「抽象」與「實作」分離：
1. **定義端口 (Port)**：在底層的 `common` 模組中，定義好抽象介面（如 `SessionRevoker`, `PasswordPolicyProvider`）。`common` 模組不依賴任何具體實作。
2. **依賴抽象**：其他模組（如 `asset`, `audit`）只需要依賴 `common` 中的**介面**，而不需要知道 `auth` 模組的存在。
3. **提供適配器 (Adapter)**：`auth` 模組作為基礎設施，負責實作這些介面，並將請求委派給自己內部的具體服務。

---

### 3. 為什麼集中在 `auth.port` 這個 Package？

既然這些 class 是 `auth` 模組對外的「適配器」，將它們集中放在 `auth.port` 這個 package 有以下幾個強烈的好處：

#### A. 語意明確，劃清邊界
當開發者打開 `auth` 模組時，可以一目了然地看出邊界：
* `auth.controller`：對外的 HTTP API 入口。
* `auth.service`：`auth` 內部的核心業務邏輯。
* **`auth.port`：`auth` 模組對其他模組提供的「適配器出口」**。
這明確告訴團隊：「這個 package 裡的 class 不是給 auth 內部用的，而是專門為了注入到 `common` 或其他模組中使用的橋接層。」

#### B. 集中管理依賴注入
這些 class 都標註了 `@Component`，Spring Boot 啟動時會掃描它們並註冊為 Bean。將它們集中管理，未來如果要進行**依賴注入的審查**、**Mock 測試的替換**，或是**模組拆分**，都能在這個 package 中一次性處理，不會散落在各個 service 中。

#### C. 遵循「依賴方向」的視覺化
在乾淨架構中，依賴方向必須指向內部（高層模組）。
* `common` 定義介面（高層）。
* `auth.port` 實作介面（低層）。
將實作放在名為 `port` 的 package 中，是一種強烈的視覺提示，提醒開發者：**「這裡是依賴反轉的發生地」**。

---

### 總結圖解

您可以把這個架構想像成「插座與插頭」：

1. **`common` 模組** 定義了牆上的**插座規格**（介面：`SessionRevoker`）。
2. **`asset` 等其他模組** 的電器只需要知道插座規格，就能插入使用（依賴抽象）。
3. **`auth` 模組** 是發電廠，它內部有複雜的發電機（`UserSessionService`）。
4. **`auth.port` 這個 package** 裡面的 class（`AuthSessionRevoker`），就是**變壓器/轉接頭**。它一頭接住發電廠的電，另一頭做成符合插座規格的插頭，提供給外面的電器使用。

將所有「轉接頭」集中放在 `auth.port`，就是為了讓系統架構保持**低耦合、高內聚**，這是企業級多模組專案中非常成熟且必要的設計！
