### 🎯 一句話總結這個架構的目的
**確保 A 租戶的使用者，絕對無法透過任何 API 或程式碼漏洞，看到或修改到 B 租戶的資料。**

---

### 🏢 白話文比喻：共構大樓的資安防線

想像您的系統是一棟 **「多租戶共構大樓」**：
1. **`TenantContext` (通行證)**：每個進入大樓的人，胸前都會掛著一張「動態通行證」。上面寫著你是「A公司員工」、「B公司員工」，或者是「大樓總管理員 (System)」。這張通行證是跟著你（當前執行緒 Thread）走的。
2. **`TenantFilterAspect` (自動感應保全)**：每間辦公室的門口都有一位保全。當你試圖進入（呼叫 Repository 查詢資料庫）時，保全會檢查你的通行證。
3. **`package-info.java` + Hibernate Filter (電子門鎖)**：這是門鎖的底層機制。保全確認你的身分後，會啟動門鎖的過濾機制，確保門打開後，你**只能**看到屬於你公司的物品（資料）。

### 🔄 完整請求生命週期 (Step-by-Step)

當一個使用者發出 API 請求時，系統會依序發生以下 5 個步驟：

#### **步驟 1：請求進入，發放通行證 (Web Filter / Interceptor)**
* 使用者帶著 JWT Token 請求 API。
* 系統驗證 Token 後，知道他是「租戶 A」的使用者。
* 系統呼叫 `TenantContext.setCurrentTenantId("tenant_A")`，將這個 ID 存入當前執行緒的 `ThreadLocal` 中。（這就是發放通行證）

#### **步驟 2：業務邏輯執行 (Service Layer)**
* 請求進入 Service 層，執行業務邏輯（例如：`orderService.getMyOrders()`）。
* Service 層準備呼叫 Repository 去資料庫撈資料。

#### **步驟 3：保全攔截與安全檢查 (`TenantFilterAspect`)**
* 因為您使用了 Spring AOP，在 Repository 方法真正執行**前**，`TenantFilterAspect` 會攔截這個動作。
* **保全開始檢查通行證 (`TenantContext`)：**
  - **情況 A (一般租戶)**：發現 `tenantId` 是 "tenant_A"。保全放行，並告訴 Hibernate：「請啟用 `tenantFilter`，參數設為 'tenant_A'」。
  - **情況 B (系統管理員/排程)**：發現是 `SystemContext`。保全會進行**交叉檢查**：如果你是透過程式碼合法標記的 (`isTrustedSystemContext`)，直接放行；如果你是手動切換的，保全會嚴格檢查你是不是 `ROLE_SUPER_ADMIN`，不是就報警（拋出 Exception）。
  - **情況 C (無通行證)**：發現 `tenantId` 是 null。保全直接鎖死（Fail-closed），拋出 Exception，防止資料外洩。

#### **步驟 4：資料庫執行，自動加上條件 (Hibernate)**
* Hibernate 收到指令，準備生成 SQL。
* 因為 `package-info.java` 定義了全域規則，且 Entity 上有標註 `@Filter`，Hibernate 會**自動**在您的 SQL 語句末尾加上：
  `WHERE tenant_id = 'tenant_A'`
* 資料庫只回傳 A 租戶的資料，完美隔離。

#### **步驟 5：請求結束，銷毀通行證 (Cleanup)**
* API 回應完畢，準備將結果傳回給使用者。
* 系統必須呼叫 `TenantContext.clear()`，清除當前執行緒的 `ThreadLocal` 資料。
* *(重要：如果不清除，下一個使用同一個執行緒的請求，可能會「撿到」上一個人的通行證，導致嚴重的資安漏洞！)*

### 🛡️ 三大核心組件職責對照表

| 組件名稱 | 所在層級 | 核心職責 (一句話) | 關鍵機制 |
| :--- | :--- | :--- | :--- |
| **`TenantContext`**<br>*(新版 common 模組)* | 基礎設施層 | **記住「當前是誰」** | 使用 `ThreadLocal` 儲存 `tenantId` 或 `System` 標記。 |
| **`TenantFilterAspect`** | AOP 攔截層 | **檢查「能否查詢」並啟動過濾** | 在 Repository 執行前攔截，驗證權限，呼叫 `session.enableFilter()`。 |
| **`package-info.java`** | 資料庫映射層 | **定義「過濾規則」** | 告訴 Hibernate `tenantFilter` 需要一個 `tenantId` 參數，以便生成正確的 SQL。 |

### 💡 常見特殊場景解析 (為什麼程式碼要寫得這麼複雜？)

您可能會疑惑，為什麼 `TenantContext` 裡要有 `runInSystemContext` 和 `isTrustedSystemContext` 這麼複雜的設計？

* **場景 1：超級管理員要看所有租戶的報表**
  管理員沒有特定的 `tenantId`。系統會將他標記為 `SystemContext`。`TenantFilterAspect` 看到後，**不會**啟用 `tenantFilter`，因此 SQL 不會加上 `WHERE tenant_id = ?`，管理員就能查到全庫資料。
* **場景 2：後台排程任務 (如：每天凌晨清理過期資料)**
  排程任務沒有 HTTP 請求，也沒有登入的使用者。它必須使用 `TenantContext.runInSystemContext(() -> { ... })`。這個方法會暫時將上下文設為 System，並且標記為 `Trusted`（受信任）。這樣 Aspect 就不會因為找不到 `ROLE_SUPER_ADMIN` 而阻擋排程執行，執行完畢後還會自動清理，非常安全。
* **場景 3：防止駭客偽造身分**
  如果一個一般使用者想辦法透過某種漏洞，讓系統執行了 `TenantContext.setSystemContext()`，因為這**不是**透過 `runInSystemContext` 進來的，`isTrustedSystemContext()` 會是 `false`。此時 Aspect 會嚴格檢查他的 Spring Security 權限，發現他不是 SUPER_ADMIN，就會直接拒絕查詢，堵住資安漏洞。
